# 跨文节表达模块完整设计与迭代指南

> 2026-07-12 后续重构以 [表达与文节重构追踪](expression_and_bunsetsu_redesign.md) 为准。本文保留现有实现、历史决策和既有 P0–P6 规划；其中“先切文节，再以 `lexical_unit` 回并”的执行顺序已被新设计替代。

## 2026-07-12 类型化统一重构

跨文节表达、词典固定表达、词法过切分和非连续呼应现已通过统一的表达分析出口处理。候选来源仍可独立维护，但正式结果共享类型、优先级、边界行为、置信度和冲突消解契约。

### 表达分类与默认优先级

| 类型 | 默认优先级 | 边界行为 | 典型对象 |
| --- | ---: | --- | --- |
| `lexical_unit` | 90 | `merge_lexical_unit` | `マジックミラー`、`しらばっくれる` |
| `idiom` | 70 | `annotate_only` | `口を開く` |
| `grammar_construction` | 60 | `annotate_only` | `〜ていい`、`にもかかわらず` |
| `correlative` | 40 | `annotate_only` | `絶対に〜ない`、`一体〜か` |

用户可以在实例规则编辑器中调整优先级。只有词汇单位允许改变边界；其他类型即使范围跨越多个文节，也只能附加语义注解。

### 词性与组合约束

用户规则现在持久化以下结构证据：

- 辞书形序列；
- IPADIC 四级词性；
- 活用类型和活用形；
- 固定词形、仅词性槽位或任意文节状态；
- 文节内部前缀、后缀或完整对齐方式；
- 连续关系或非连续前后锚点。

规则复用时必须同时满足已配置的词性、活用和组合范围。词典候选也必须先通过组合方式检查：以助词或助动词起始的普通句法拼接不会仅因词典命中而成为正式表达；含助词的候选只能按固定惯用语方向判定，不能合并为词汇单位。

### 统一解析和边界重组

`Engine` 使用单一表达分析出口依次汇集用户、内置、词典和非连续候选，再按优先级、置信度及范围进行同类排重。`merge_lexical_unit` 在最终出口重组 `AnnotatedToken` 和 `Bunsetsu`，同时保留范围内全部底层 morpheme，供双击拆分和局部查询。

发生词汇边界规则新增或删除后，阅读器重新运行正式分析，不再使用依赖旧 token 数量的局部注解刷新。

### 非连续边界修复

扁平语素流不再删除标点和换行，而是保留为边界节点。非连续呼应遇到句号、问号、感叹号、换行或其他非内容边界时终止，并优先选择最近的相容尾项，避免跨句、跨段和跨对话连接。

### 实例规则编辑器

正文范围拖选仍是规则创建入口。改进后的 modal 支持：

- 选择词汇单位、固定惯用语、语法构式或非连续呼应；
- 编辑规则标签和整体说明；
- 调整优先级并查看类型决定的边界行为；
- 按文节配置固定词形、仅词性槽位或任意匹配；
- 选择文节内部实际参与匹配的语素；
- 查看四级词性、活用类型、活用形和最终规则签名；
- 仅在非连续呼应类型中插入有限间隔。

表达标签可点击并以整体表层形发起词典查询；合并后的词汇单位仍保留内部语素，双击具体语素会以该语素自身的辞书形和读音下钻查词。原“拆为形态素”操作已移除；词条操作和 N-best 分词候选改由右键菜单打开。

### 下一版本范围

词典正文中的品词标记、活用标记、惯用句结构、多词典重复、多重候选和空导航词条尚未接入自动规则证据。本版本仍以形态素结构和组合约束过滤词典候选；词条详情证据及候选消歧留待下一版本实现。

## 1. 模块目的

本模块识别和保存跨越两个以上文节、但整体具有固定或半固定含义的日语表达。

典型对象：

- `有無を｜言わさず`
- `されていたのにも｜かかわらず`
- `兄に｜負けるところが｜多い`
- `食料に｜すぎない`
- `何が｜何でも`

模块不负责以下对象：

- 已经完整位于一个文节内的语法，如当前 Pipeline 中的 `ない限り`、`からには`。
- 普通的任意相邻文节组合。
- 改写 Vibrato lattice 中的原始形态素节点；词汇单位只在派生分析层重组 token 和文节。
- 已能在单一文节内由原有词典边界层解决的词头，如 `警察署`、`はぐれ者`。
- 普通长程依存；非连续层只识别有显式前后锚点和有限间隔的呼应规则。

核心原则是：底层 morpheme 和字符范围保持可追溯；只有 `lexical_unit` 可以重组派生 token 和文节，其他类型只增加语义关系。

## 3. 数据模型

### 3.1 `ExpressionAnnotation`

每次实际命中都会向相关 `AnnotatedToken.expressions` 附加注解：

- `match_id`：一次命中的稳定标识。
- `rule_id`：用户 SQLite 规则 ID；内置规则使用负数 ID。
- `label`：正文旁显示的短标签。
- `description`：整体含义或使用条件，不是逐词翻译。
- `origin`：`builtin` 或 `custom`。
- `expression_type`：词汇单位、惯用语、语法构式或非连续呼应。
- `priority`、`confidence`：统一冲突消解依据。
- `boundary_effect`：合并词汇单位或仅注解。
- `position`：`start`、`middle`、`end` 或 `single`。
- `token_range`：半开 token 范围。
- `char_range`：半开字符范围，可从文节内部的语素开始或结束。
- `surface`：本次实际匹配到的表层文本。

`token_range` 用于跨胶囊连接；`char_range` 用于只标记真正属于表达的语素。例如 `兄に｜負けるところが｜多い` 的表达字符范围从助词 `に` 开始，不包含参数 `兄`。

### 3.2 用户规则

`ExpressionRule` 包含：

- `label`、`description`、`origin`、`created_at`。
- `expression_type`、`priority`、`boundary_effect`。
- 一个或多个 `ExpressionPatternPart`。

每个 `ExpressionPatternPart` 对应一个文节：

- `lemmas`：该文节结构语素的辞书形序列。
- `pos`、`pos_details`：一级及四级词性序列。
- `conjugation_types`、`conjugation_forms`：活用约束。
- `surface_hint`：只供用户理解，不参与结构相等判断。
- `is_slot`：是否为可变槽位。

槽位不等于无约束通配。槽位忽略具体辞书形，但仍要求词性与助词序列相同。例如 `{名詞＋を}` 可以从 `何を` 复用到 `仕事を`，不会匹配 `仕事が`。

## 4. 正式分析流程

当前顺序如下：

1. 去除并记录作者振假名。
2. Vibrato 形态素分析。
3. 文节组块。
4. 文节内语法匹配。
5. 应用已选择的 N-best 文节内语素方案。
6. 外部词典校正词汇边界。
7. 用户画像评分。
8. 统一汇集用户、内置、词典和非连续表达候选。
9. 按类型、优先级、置信度和范围消解冲突。
10. 应用词汇单位边界重组。
11. 记录曝光。

历史 `user_merge_rules` 不再进入正式 Pipeline。旧表和 API 暂时保留用于兼容旧数据；新的边界调整只来自类型明确的 `lexical_unit` 表达规则。

## 5. 内置连续表达状态机

### 5.1 输入展开

状态机把全部 `AnnotatedToken` 展开为连续语素流。每个扁平语素保留：

- 所属 token 索引。
- 表层形与辞书形。
- 主词性。
- 全局字符范围。

空白语素被跳过，标点仍保留，因此模式不会无意跨过标点。

### 5.2 atom 匹配

目录中的一个 atom 可指定：

- 一个或多个允许的辞书形。
- 一个或多个允许的表层形；仅在需要区分同一辞书形的活用或助词形式时使用。
- 可选主词性。
- 省略辞书形时，成为受词性约束的槽位。例如任意动词接 `間もなく`。

所有 atom 必须在扁平语素流中连续匹配。只有命中跨越至少两个 token 时，才作为跨文节表达返回。

### 5.3 范围重建

状态机使用首末命中语素计算精确 `char_range`，再根据所属 token 计算 `token_range`。因此模式可以从一个文节内部的助词开始，而 UI 仍能连接整个涉及的文节。

### 5.4 内置目录格式

```json
{
  "id": "concessive_nimo_kakawarazu",
  "label": "にもかかわらず",
  "description": "前项事实成立，却出现与通常预期相反或不受其影响的后项。",
  "category": "让步",
  "atoms": [
    { "lemmas": ["に"], "pos": "助詞" },
    { "lemmas": ["も"], "pos": "助詞" },
    { "lemmas": ["かかわる", "関わる"], "pos": "動詞" },
    { "lemmas": ["ぬ"], "pos": "助動詞" }
  ]
}
```

辞书形变体必须来自实际 Pipeline 观察。例如 IPADIC 把 `言わさず` 分析为 `言わす＋ぬ`，目录同时允许必要的变体，但不应用宽泛表面模糊匹配替代语法判断。

### 5.5 基于词典的跨文节固定表达扫描

对于结构固定且没有可变槽位的谓宾或其他简单词汇组合（例如 `口を開く`、`足を取られる`、`手に入れる` 等），若手工在内置目录中维护则过于繁冗，而这些惯用语往往已经收录在本地词典中。系统设计了文节级别的词典固定表达扫描机制：

1. **文节滑动**：以文节数量 $N$ 作为滑动窗口大小（在 `2..=4` 范围内从大到小滑动）。相较于在扁平语素流上盲目滑动，这确保了窗口天然对跨越的文节数量敏感，且匹配出的区间一定是跨文节的。
2. **端点穷举与语素拼接**：对于滑动窗口 `tokens[s..=e]`，穷举起点文节 $T_s$ 中的语素偏移量 $p$ 和终点文节 $T_e$ 中的语素偏移量 $q$，提取子序列并进行拼接。
   - **拼接策略**：为兼容各种活用的惯用语（如 `口を開いた`、`足を取られる`），前 $len - 1$ 个语素使用其表层形（`surface`），最后一个核心词语素使用其辞书形（`base_form`）还原。
3. **词典精准检索**：拼接成的查询词将调用 `DictionaryEngine::contains_exact` 在本地词典中进行检索。
4. **长词优先与范围排重**：
   - 匹配机制优先尝试更大文节跨度 $N$ 的匹配。
   - 若匹配成功，其字符区间将被记录至 `matched_char_ranges`。后续任何子区间的短词匹配如果被完全包含在已匹配的更大区间内，则会被跳过，从而保证只呈现最长最完整的固定表达。
   - 被匹配的 token 会被注入 `rule_id = -9999`，`origin = "dictionary"` 且 label 为词典词头的 `ExpressionAnnotation`。

### 5.6 非连续呼应表达（NFA）匹配

对于首尾词固定、中间可变的跨文节语法呼应现象（例如 `まるで……ようだ`、`一体……か`、`たとえ……ても`），系统在原有连续表达状态机之上设计并集成了非连续呼应表达匹配层：

1. **结构化规则定义**：在内置目录 `expression_patterns.json` 中定义 `head_atoms`（前项固定锚点）与 `tail_variants`（后项变体列表，匹配其一即可），并限制其跨越的文节数 `gap_bunsetsu`（默认为 `[0, 10]`）。
2. **多变体深度匹配**：在扁平语素流上首先定位所有的前项命中。对每一个命中，在其后的 `gap_bunsetsu` 范围内扫描符合 `tail_variants` 中任一形态的后项组合。
3. **分词自适应调节**：针对连语 `としても` 在 IPADIC 下切分为 `として`＋`も` 的特殊分词，以及口语 `たって`/`だって` 被切分为单语素的特例，匹配算法通过扩充变体树实现自适应的精确容错。
4. **句子边界截断**：若前项与后项之间出现了句号（`。`）、问号（`？`）、感叹号（`！`）或硬换行符，为防止无节制的越界匹配，匹配器会自动舍弃并截断当前扫描。
5. **长匹配优先与包含排重**：对同一个前项，若有多处后项被匹配，优先采用跨越距离最长（`end_token` 最大）的一个匹配；随后执行全范围包含性检查，自动滤除短距离的冗余子区间。
6. **注解输出**：匹配成功后，仅在首尾两个锚点所在的 token 上分别标记 `start` 和 `end`（中间 token 标记 `middle`），而 `token_range` 和 `char_range` 依然保留完整区间。这确保了在 UI 渲染上既可以完整绘制跨文节的连接带，又不会把中间完全自由的填充词误标为语法本体。

## 6. 用户规则匹配

### 6.1 规则创建

阅读页横向拖过两个以上胶囊后打开编辑器。用户必须确认：

- 短标签。
- 整体含义或使用条件。
- 哪些文节是可变槽位。

保存时从每个文节提取辞书形和词性签名，并写入 `user_expression_rules`。

### 6.2 结构归一化

为了跨活用复用，签名进行有限归一化：

- `た`、`ます` 视为时态或礼貌尾，不限定表达模板。
- `だ`、`です` 统一为 `<copula>`。
- 句末 `か`、`ね`、`よ` 不限定模板。
- 否定等会改变意义的形式继续保留。
- `surface_hint` 永不参与匹配。

该归一化只服务人工规则，不替代完整日语语法分析。

### 6.3 重叠规则

当前允许多个表达重叠。这样可以同时观察较长句式和其中的短固定表达。UI 默认以首个注解决定连接带主样式，但模型保留全部注解。

后续若出现视觉拥挤，应增加显示优先级，而不是在核心层丢弃重叠命中。

## 7. SQLite 结构

```sql
CREATE TABLE user_expression_rules (
    id           INTEGER PRIMARY KEY AUTOINCREMENT,
    label        TEXT NOT NULL,
    description  TEXT NOT NULL DEFAULT '',
    origin       TEXT NOT NULL DEFAULT 'custom',
    pattern_json TEXT NOT NULL UNIQUE,
    created_at   TEXT NOT NULL DEFAULT (datetime('now'))
);
```

相同 `pattern_json` 再次保存时更新标签和说明，不生成重复规则。开发期旧表缺列时，初始化过程会执行兼容迁移。

## 8. UI 逻辑

### 8.1 创建

- 拖拽只能在同一段落内进行。
- 已知词可以参与表达；标点不能作为拖拽端点。
- 松开后不立即保存，而是进入表达编辑器。
- 保存后重新分析原文，使自动匹配结果成为正式后端输出。

### 8.2 展示

- 文节胶囊保持原大小。
- 涉及的 token 下方使用细连接带。
- 真正命中的语素使用紫色点状下划线。
- 首 token 显示短标签，悬停标题显示整体说明。
- E-ink 模式改用黑色细线和边框标签。

### 8.3 管理

顶部“表达式”入口打开规则侧栏。侧栏显示标签、原始选择提示、槽位和整体说明，并支持删除。删除后当前文本重新分析，旧注解立即消失。

## 10. 当前真实文本证据

第一话全章当前命中 10 次、9 种表达：

- `有無を｜言わさず`
- `休む｜間もなく`
- `食料に｜すぎない`
- `にも｜かかわらず`
- `こと｜なく`，在两个不同句子中复现。
- `に｜しては`
- `何が｜何でも`
- `目も｜くれず`
- `なすが｜ままに`

这些命中用于证明模块可工作，不表示目录已经覆盖全部跨文节表达。

## 11. 添加内置表达的判定标准

新增目录项前应依次确认：

1. 整体是否具有稳定意义或关系功能，而非任意相邻短语。
2. 当前 Vibrato/IPADIC 是否确实将其分到两个以上文节。
3. 表达本体从哪个语素开始，前接名词是否只是参数。
4. 辞书形是否存在 IPADIC 特有分析，需要哪些实证变体。
5. 是否连续。非连续表达不要强塞进当前连续状态机。
6. 是否会与更短规则重叠，重叠是否有解释价值。
7. 至少在真实上下文中用 `expression-preview` 和 `expression-scan` 人工阅读。

## 12. 已知限制

- 自动词典候选目前只使用词头存在性和文本侧组合结构，没有读取词条正文中的品词、活用、惯用句标记和义项类型。
- 词典查询可能返回同形异义、同读异表记、多词典重复、跳转或空内容候选；当前表达注解没有绑定确定的 `entry_key`。
- 自动词汇单位分类仍是有限启发式，没有与 Vibrato lattice/N-best 做联合全局评分。
- `merge_lexical_unit` 重组派生 `AnnotatedToken` 和 `Bunsetsu`，但不会向 Vibrato 系统词典写入新节点，也不会改变原始 lattice 成本。
- 用户规则会保存实例中的四级词性和活用信息；尚未提供逐字段的“严格、归一化、忽略”开关，某些活用变化可能因此匹配过窄。
- 组合方式验证目前是有限日语规则，不是完整依存分析；字面自由组合与惯用义并存时仍可能需要人工判定。
- 冲突解析已经使用类型、优先级、置信度和范围，但尚未实现完整 span graph；复杂交叉重叠仍可能只在逐 token 层排重。
- 非连续匹配保留硬边界并选择最近尾项，但不支持小句依存、命名捕获、跨锚点一致性和嵌套状态。
- 表达整体查词当前按标签或实际表层重新查询，未直接打开命中时确认的词典条目。
- 当前反馈能力以新增、删除用户规则为主，尚无“正确、类型错误、范围错误、不是表达”的上下文级负反馈。
- 内置规则尚未统一提供版本、来源、正反例、可信度、禁用状态和 schema 迁移工具。

## 13. 按原始能力要求推导的后期任务

后续工作必须持续覆盖最初提出的三类对象，而不能只优化总命中数：

1. 真实跨文节的组合表达：保持文节边界，提供整体语义和词典/语法说明。
2. 文节划分过细的真实词汇：形成词汇单位并影响派生分词和文节，同时允许下钻内部语素。
3. 词典或用户规则误匹配：给出可解释的拒绝原因，并能通过上下文反馈避免复发。

### 13.1 P0：真实文本基线与回归验收

- 完成第一话 341 项的类型化人工审计，标签至少包括 `lexical_unit`、`idiom`、`grammar_construction`、`correlative`、`false_positive`、`wrong_range` 和 `wrong_boundary_effect`。
- 固定正例：`マジックミラー`、`しらばっくれる`、`口を開く`、`〜ていい`。
- 固定反例：普通句法中的 `につく`、`とする`，以及跨对话的 `絶対に〜ない`、跨多句的 `一体〜か`。
- CLI 报告分别统计候选召回率、各类型准确率、边界决策准确率、整体释义可用率和硬边界越界数；硬边界越界目标必须为 0。
- 为每次误判保存候选来源、形态素签名、拒绝/接受证据、最终类型和边界行为，避免仅保存二值结论。
- 建立小型快速回归集和章节级离线审计集；日常提交运行前者，目录或算法调整后运行后者。

### 13.2 P1：词典结构证据与多候选消歧

- 从词条正文提取品词、活用类型、词条类型、惯用句结构和规范词头模式，建立可重建的结构化索引；不要在每次阅读分析时临时解析完整 HTML。
- 区分精确表记、读音、跳转、模糊命中和空内容结果；跳转或清洗后空条目不能作为自动规则正证据。
- 将多词典重复按“规范词头＋类型＋词性＋活用”归并为证据签名，不按返回条目数量重复加权。
- 多个不同结构签名同时相容时保持未决，不使用“任一候选相容”自动通过；modal 提供候选条目选择，并保存确定的 `entry_key`。
- 用文本侧 morpheme 结构验证词条侧结构：例如 `に/助詞＋つく/動詞` 不得被单一动词 `似付く（動カ五）` 证明；`と/助詞＋する/動詞` 不得被 `賭する（動サ変）` 证明。
- 整体释义直接打开命中时绑定的词典条目；词典条目缺失时明确回退到规则说明，不伪装成词典释义。
- 验收：抽样覆盖单一候选、同形异义、多词典同义重复、读音候选、跳转词条、空内容词条和无词条语法构式。

### 13.3 P2：组合方式 DSL 与日语句法深化

- 把当前有限组合判断升级为统一规则 DSL，显式描述相邻、同词汇单位、同文节、下一文节、有限间隔、同小句和依存关系。
- 为规则原子提供表层、辞书形、读音、四级词性、活用类型、活用形、自立/非自立、接头辞/接尾辞约束。
- 增加字段级匹配策略：`exact`、`normalized`、`class_only`、`ignore`；默认策略按表达类型设定，避免固定规则过窄或槽位过宽。
- 建立轻量级小句和论元结构：格助词前项、谓词核心、引用结构、补助用言、接辞附着和复合名词连接。
- 对 `する`、`なる`、`つく`、`ある`、`いる` 等高频轻动词设置更严格的结构模板，禁止从字符串存在性推导固定表达。
- 区分字面义和惯用义共存；无法由结构判定时降低置信度并交给用户，而不是自动改写边界。
- 长期可接入依存分析器，但输出仍需转换为统一 `SpanAnalysis`，不能形成第二套表达模型。

### 13.4 P3：词汇边界、N-best 与文节联合决策

- 在文节最终确定前生成 `LexicalCandidate`，联合使用 lattice 路径成本、词典结构证据、接辞连接和用户选择评分。
- 只有 `lexical_unit` 参与词汇边界选择；惯用语、语法构式和呼应结构不得改变 lattice 或物理词边界。
- 当 lattice 中存在整体路径时优先选择真实节点；不存在整体节点时允许派生词汇单位，但必须保留来源语素和可逆映射。
- 重新计算受影响的局部文节、词头、画像评分和曝光对象，避免只修改显示胶囊而留下内部模型不一致。
- 支持用户在整体词汇和内部细分之间持久切换；双击语素只下钻本次查看，不应无意覆盖全局边界选择。
- 处理表达嵌套后的 token/char 范围映射，使用全局 span graph 解决包含、交叉、互斥和可共存关系。
- 验收：文本可逆重建率和字符范围完整性保持 100%，用户撤销规则后恢复原始分析，画像曝光不重复计算。

### 13.5 P4：非连续状态机和边界域

- 将边界从单一布尔值升级为类型化事件：弱标点、强句末、硬换行、段落、引号开闭、对话角色切换和 Markdown 块边界。
- 每条规则配置允许跨越的边界类型、最大 morpheme 数、字符数、文节数和小句数。
- 默认选择最近且句法相容的尾项；多尾项、嵌套呼应或共享尾项进入 span graph 决策。
- 支持命名捕获、锚点角色、跨 part 一致性和有限可选分支。
- 跨句或跨段能力只能由明确规则开启，并要求真实语料正反例；不得恢复无约束长程扫描。

### 13.6 P5：用户反馈与规则完整编辑

- modal 增加词典候选绑定、字段级约束开关、gap 上下限、边界类型、置信度阈值和正反例预览。
- 保存前用当前实例执行规则预演，显示实际锚点、将被合并的文节、被忽略的语素和验证警告。
- 提供“正确”“这是一个词”“是表达但不要合并”“类型错误”“范围错误”“不是表达”和“选择其他分词”。
- 反馈作用域必须明确为当前上下文、当前表层、lemma/POS 签名、特定词条或全局规则。
- 负反馈保存为结构化抑制条件，不能把一次 `指先についた` 的拒绝转换成全局字符串黑名单 `につく`。
- 支持编辑、复制、禁用、调整优先级和查看规则命中历史，不再只有新增和删除。
- 整体卡片同时提供规则说明、绑定词典释义、实际捕获内容和内部语素入口。

### 13.7 P6：目录、画像、导出与性能治理

- 为内置和用户规则统一增加 `schema_version`、`rule_version`、`source`、`examples`、`counter_examples`、`confidence`、`enabled` 和变更时间。
- 提供 JSON schema 校验、导入导出、兼容迁移和失效规则诊断。
- 按词汇单位、惯用语、让步、限定、比较、时间、许可、推量等分类浏览。
- 表达熟悉度、曝光和用户笔记使用独立画像实体，不与单词曝光次数混合。
- 导出同时保存整体表达、内部词汇、上下文锚点、整体释义来源和用户选择的边界。
- 缓存词典结构证据和规则编译结果；章节分析采用批量查询，避免逐候选解析 HTML 或往返 SQLite。
- 建立候选数、各阶段耗时、规则命中率、冲突数和拒绝原因分布的诊断指标。

### 13.8 实施依赖与完成定义

推荐依赖顺序为：P0 基线 → P1 词典证据与 P2 组合 DSL → P3 联合边界 → P4 非连续深化 → P5 完整反馈 → P6 治理。P1 与 P2 可以并行研究，但在没有 P0 基线前不得用命中数量宣称准确率提升。

模块达到完整目标需要同时满足：

- 三类原始问题均有独立类型、处理策略和回归样本；
- 所有自动规则都经过词性、活用、组合方式和边界验证；
- 词汇单位能够影响派生分词和文节，其他表达不会错误合并；
- 整体释义可追溯到确定词条或确定规则来源；
- 用户可以从文本实例完整配置、预演、修正和撤销规则；
- 多候选和冲突结果可解释，不以任意首项或最长项静默决策；
- 跨句、跨段和跨对话误连在默认规则下为 0；
- 文本重建、字符范围和撤销恢复保持机械完整。

## 14. 维护注意事项

- 不要重新启用旧文节合并规则来实现表达。
- 不要用整句正则直接操作原始 Markdown；先使用 ruby 预处理和正式 Pipeline 范围。
- 不要让 `surface_hint` 参与匹配。
- 不要用机械命中数量声称语义正确率。
- 修改目录后优先用实际小说行进行交互阅读，并记录误匹配上下文。
