# Kotoclip v1.0 缺口审计与模块补齐方案

状态：执行草案（2026-07-19）

范围：当前 `master` 相对 v1.0 产品闭环的剩余工作、模块边界、规模、覆盖、目标、依赖、提交拆分、验证和实验方法。

本文不替代各语言模块的权威协议。语法以 `grammar_morphology_and_functional_pipeline.md` 为准，词典以 `dictionary_lookup_and_bubble_refactor.md` 和 `dictionary_internal_architecture.md` 为准，表达以 `cross_bunsetsu_expressions.md` 和 `expression_and_bunsetsu_redesign.md` 为准，增量管线以 `incremental_pipeline_roadmap.md` 为准。

## 1. 结论

按可以独立设计、提交和验收的工作包计，v1.0 还剩 **8 个模块包**：

1. 统一文本现象协议与 v1.0 评测基线；
2. N1～N5、语法、构词、文节、表达和词典质量的覆盖收口；
3. 本地文档库、正式导入入口与发布资源边界；
4. 句法提供者与基础读解解释；
5. 语义学习事件、知识聚合、可重算画像与完整导出；
6. 动态卡片、复习队列与 FSRS；
7. 自适应阅读辅助、阅读模式与密度预算；
8. BYOK AI 上下文问答、人工授权和降级。

其中前两个模块是在现有分析基础上收口，后六个模块大多处于未实现或早期框架阶段。P8 同步与 SaaS 明确属于 v1.0 之后，不计入上述 8 项。

外部库或深度分析能力进入候选清单，不表示已经决定引入。它们作为模块内实验决策门：结论只能是“正式引入”“仅作为可选／研究 provider”“不引入或延后”。

不建议把当前状态压缩成单一百分比。语言分析主干的工程成熟度已经较高，但“导入文本 → 连续阅读 → 记录困难 → 聚合知识 → 生成卡片 → 复习 → 调整下一次辅助”的产品闭环仍未形成。若必须区分，可描述为：

- 分析与解释基础设施：约 70%～80%；
- v1.0 用户闭环：约 25%～35%；
- 发布级覆盖、实验和数据治理：约 20%～30%。

## 2. 审计依据

### 2.1 当前已形成的强基础

- `DocumentSession`、revision、稳定 Token ID、范围 Patch、渐进加载、暖缓存和领域 mutation 已完成 P1～P6 重构。
- Vibrato fork 已提供真实 lattice N-best，用户选择能够持久化并触发局部重算。
- 语法 G0～G5 已闭环：105 个 concept、154 个 sense、112 个 realization/rule、105 份讲解，目录审计和正文精确讲解解析可用。
- 多词典统一层已覆盖大辞林、小学馆和 Crown 三个适配器，建立 occurrence、sense tree、查询证据、候选与单活动词典气泡。
- 表达层已有连续、非连续、用户规则、规则预演、类型化候选和精确范围。
- 用户画像已有曝光、已知／未知、汉字推断、分词选择和词典选择等局部状态。
- 当前导出能够生成带哈希、上下文、词典释义和用户笔记的选中词汇 JSON。

### 2.2 当前不能算作 v1.0 完成的部分

- 语法 concept 当前只分布在 N3、N4、N5 和未知等级；N1、N2 和统一目标目录分母缺失。
- 现有 95.73% 指标仅是第一话词汇出现级词典查询覆盖，不代表词义、边界、语法或解释准确率。
- 当前主界面仍以粘贴文本为正式入口，没有文档库、章节、阅读位置和导入适配器。
- 代码中没有统一 `TextPhenomenon`、`PhenomenonOccurrence`、`learning_events`、`review_logs`、知识项、卡片或 FSRS 状态。
- 当前没有 `SyntaxAnalysisProvider`、GiNZA/KWJA 接入、正式小句／依存基线或读解解释模块。
- `master` 没有阅读模式策略文件；2026-07-17 的 `60bbf0d` 仅存在于 `backup-master`，只能作为原型参考。
- LLM 词典消歧只有离线 schema、强校验和宿主端口，没有 HTTP transport、Tauri 命令、凭据、UI 或真实评测。
- 当前 Insider 资源与公开发布资源边界尚未彻底分开，公开 v1.0 不能默认携带权利未确认的研究词典。

## 3. 从既有模块提炼的设计与提交经验

### 3.1 语法模块提供的经验

`dc2b6eb feat: 完成语法知识识别与讲解模块` 一次覆盖 60 个文件，新增约 111,231 行、删除 412 行。它完成了作者源、schema、编译产物、运行时、CLI、Tauri、Vue、代表句、章节基线和文档的垂直闭环。

有效做法：

- 先固定 concept／sense／realization／rule／explanation 的稳定身份，再扩识别和 UI；
- 作者源与编译产物分离，构建器检查未知字段、悬空引用、模板变量和来源；
- 正文 occurrence 精确绑定讲解，不按表面字符串回退；
- CLI、代表句、章节 residual、目录审计和构建检查同时进入完成定义；
- 明确 G0～G5 工程闭环与 G6 内容扩充的边界，不把框架完成误写成 v1.0 覆盖完成。

该提交之后同日又出现讲解气泡、核验标识、内容重复、连体「な」、功能词头、活用边界和词形显示等连续修复，说明后续模块不宜把 schema、运行时、UI、海量资产和视觉细节压进一个不可复核的大提交。

### 3.2 多词典统一模块提供的经验

`654108a refactor(dictionary): unify multi-dictionary lookup and bubble` 覆盖 33 个文件，新增约 7,222 行、删除 930 行。它在提交前保留 18 词问题基线和源内容笔记，在实现中分开查询、match evidence、occurrence IR、分词典适配器、presentation、Vue renderer 和 CLI，提交后另建 P1～P3 后续清单。

有效做法：

- 先以真实样本确认系统性根因，不从某一个词的 DOM 特例直接修改架构；
- 固定公共 occurrence／sense IR，词典差异只进入适配器；
- 查询事实、选择状态和显示状态分离；
- 每个样本同时保存完整 Lookup JSON 和与真实气泡等价的 HTML 预览；
- “主要重构完成”只表示没有 P0 架构阻塞，不表示全部内容精度已经完成。

### 3.3 后续模块统一采用的提交模板

每个模块原则上拆为 5～7 个聚焦提交：

1. `docs(<module>): freeze baseline and acceptance`：问题、基线、目标、非目标、schema 草案、金标和风险；
2. `feat(<module>): add domain schema and storage`：稳定 ID、版本、迁移、来源与失效规则；
3. `feat(<module>): implement engine and cli`：核心算法、批处理、诊断、fixture 生成；
4. `feat(<module>): integrate session and ui`：Tauri DTO、前端 view model、交互与降级；
5. `test(<module>): add goldens and corpus audit`：极简正反例、真实语料、随机一致性和性能；
6. `docs(<module>): record validation and followups`：结果、限制、下一阶段清单；
7. 只有在模块确实完成时再使用总括性的 `feat:` 完成提交，不用提交标题替代验收证据。

生成资产与大型章节基线应独立提交或至少独立 diff 区块，避免淹没手写代码。UI 样式调整与语义修复分开提交。每个提交先运行聚焦检查，模块收口再运行仓库级构建和完整审计。

## 4. 规模口径与总体依赖

规模只统计手写实现、测试和文档，不把生成目录、词典内容或大体量金标 JSON 计入代码量。

| 级别 | 典型范围 | 聚焦提交数 | 单人净开发量参考 |
| --- | --- | ---: | ---: |
| M | 单一协议或 UI 垂直切片 | 3～5 | 1～3 周 |
| L | 新存储／引擎／UI／评测闭环 | 5～8 | 3～6 周 |
| XL | 数据密集或跨多个语言层的完整模块 | 8～12 | 6～10 周 |

整体估算为 45～65 个聚焦提交。单人串行实现约 24～36 个净开发周，语言标注、真实用户实验、许可确认和日历等待时间另计。语言覆盖扩充应与事件／卡片开发并行，否则会成为最后阶段的长尾。

依赖主线：

```text
M0 统一协议与评测
├─ M1 语言覆盖收口
├─ M2 文档库与发布资源
├─ M3 分析器选型实验 → 条件性句法与基础读解
└─ M4 事件与知识聚合
   └─ M5 动态卡片与 FSRS
      └─ M6 自适应阅读辅助

M7 BYOK AI 问答依赖 M0；
上下文读解能力依赖 M3，授权、缓存和审计依赖 M4。
```

关键路径为 M0 → M4 → M5 → M6。M1 是贯穿全程的发布质量主线。M2 应在 M4 前完成稳定文档、章节和位置身份，否则事件无法可靠绑定来源。

### 4.1 必须先实验再决定是否引入的项目

语言分析与文本深度：

1. 继续使用 Vibrato/IPADIC，或引入 Sudachi、MeCab＋UniDic 作为替代／第二证据源；
2. 当前文节候选规则是否已经足够，或需要 GiNZA／KWJA 提供文节边界和主辞证据；
3. GiNZA 是否只用于依存、谓词和句子主干，还是完全不进入正式包；
4. KWJA 是否只用于研究，或有限引入述语项结构、桥接照应、共指和篇章关系；
5. GiNZA 不同模型或其他 provider 的准确率、CPU、内存、模型体积和启动成本是否值得；
6. 外部 analyzer token 与当前规范字符、ruby、Vibrato morpheme 和 Bunsetsu 的对齐是否稳定；
7. 多 provider 一致性融合是否优于单 provider，冲突是否能保留候选而不改写规范 Token；
8. 句法证据是否真实降低表达误报、语法 sense 歧义和文节错误，还是只增加复杂度；
9. 句子骨架、接续、指代、省略和篇章提示分别是否提升理解，不能把“深度分析”作为一个整体一次引入；
10. 外部分析器采用内置 sidecar、可选下载、插件、仅开发工具还是完全不分发；
11. 模型／词典许可、再分发、更新、离线安装和版本锁定是否满足发布要求。

其他条件性依赖：

12. FSRS 使用现成库、移植参考算法还是先用更简单调度器；
13. AI 凭据使用系统凭据库、Tauri 安全存储插件或其他本地方案；
14. AI 网络层使用通用 HTTP／兼容协议还是 provider SDK；
15. EPUB、网页、OCR 等导入库是否延后到 v1.0 后，当前结论仍为不进入正式范围。

每项只需在对应模块文档记录候选、最小实验、结论和理由；未完成实验不得在状态表中标记为“已选型”或“已完成”。

## 5. M0：统一文本现象协议与 v1.0 评测基线

### 5.1 规模与目标

- 规模：L，5～7 个提交。
- 目标：建立所有语言模块、知识聚合和 UI 都能消费的稳定 `TextPhenomenon`、occurrence、evidence 和 artifact 协议；给 v1.0 的 90%～95% 目标提供明确分母和可重复评测。
- 非目标：本模块不增加大量语法规则，不实现用户卡片，也不要求一次解决多义现象。

### 5.2 覆盖

协议必须覆盖：

- 词汇、构词、文节、活用、功能语素、语法构式、惯用语、呼应；
- 谓词、格成分、从句、引用、接续和基础篇章关系；
- accepted、pending、rejected、unknown；
- 来源、版本、许可、置信度、证据、反证据、冲突组和人工选择；
- 规范目录项与文档 occurrence 分离；
- Rust、TypeScript、JSON 导出和持久化 schema 的同版本契约。

评测集分三层：

1. 极简正反例：每个正式规则至少一正一反；
2. 多体裁金标：轻小说／现代小说、近现代文学、新闻／说明、SRT／脚本、边界压力文本各 200 句，共 1,000 句；
3. 未标注覆盖语料：每类至少 10 万字符，用于目录分布、residual 和性能，不冒充准确率金标。

10% 金标由第二人重复标注。边界 exact agreement 目标不低于 95%，类别一致性目标 Cohen's kappa 不低于 0.85；达不到时先修订标注协议。

### 5.3 设计

- 在 core 中新增版本化 phenomenon schema 和 category registry，不让前端类型成为事实源。
- 为现有 GrammarOccurrence、ExpressionCandidate、DictionaryLexicalUnitAnnotation、BunsetsuFunctionAnnotation 建立无损映射。
- 定义 `ArtifactDescriptor`：输入指纹、上下文半径、边界变更能力、缓存策略和生产者版本。
- 建立统一 audit 输出：目录覆盖、现象召回、精确率、边界准确率、解释可用率、严重错误率和 residual 分布。
- 金标以源文本字符坐标为主，不保存易漂移的 token index 作为唯一身份。

### 5.4 验证与实验

- schema round-trip、未知字段拒绝、版本迁移和 Rust／TypeScript fixture 一致性；
- 现有语法、表达和词典基线迁移前后结果逐 occurrence 比较；
- 随机生成范围与多字节字符测试，文本重建和字符范围完整性必须为 100%；
- audit CLI 对同一输入、同一版本重复运行结果完全一致；
- 评测报告必须按体裁和 category 分层，不只输出单一总分。

完成定义：所有后续模块可以只依赖统一协议和评测入口，不再各自发明 occurrence、来源、置信度和覆盖率口径。

## 6. M1：语言目录、规则与真实现象覆盖收口

### 6.1 规模与目标

- 规模：XL、数据密集，8～12 个代码／治理提交，另有持续的 20～50 项内容批次。
- 目标：把现有“框架可用”推进到现代标准日语目标域内主要模块 90%～95% 的目录覆盖或真实现象召回，并把严重误匹配控制在 5% 以下。
- 非目标：不要求对所有多义词、助词或省略唯一消歧；无法可靠判断时保留候选或交给用户／AI。

### 6.2 覆盖

- 语法：补齐 N1、N2，校正 N3～N5；按知识族扩 concept、sense、realization、rule 和 explanation。
- 构词与文节：从当前 6 条构词、4 条词典词汇候选、10 条边界规则扩为有目标目录和反例的治理体系。
- 表达：完成第一话 341 项类型化人工审计，补词典结构证据、组合 DSL、非连续边界域和 span graph 冲突。
- 词典：完成 P1 精度项和固定回归夹具；公开报告精确表记、读音、导航、空条目、同形异义和多词典重复的分项结果。
- JLPT：等级必须保存来源和冲突，不把单一来源当作绝对难度。

内容批次沿用语法模块经验：每批 20～50 项，必须包含真实上下文、正反例、来源、规则、讲解和 audit 状态。不得只增加表面串匹配。

### 6.3 量化目标

- 目标目录覆盖：90%～95%；
- 各主要体裁目标现象召回：不低于 90%，总体目标 90%～95%；
- accepted 精确率：不低于 95%，高风险自动边界变更目标不低于 98%；
- 严重错误率：低于 5%；
- 硬边界越界：0；
- 文本重建和字符范围完整性：100%；
- accepted occurrence 到有效解释或确定词条的可追溯率：100%；
- unknown／pending 必须有原因分布，不计为覆盖成功。

### 6.4 验证与实验

- 每次内容批次运行 grammar library audit、代表句、章节 residual 和多体裁抽样；
- 表达层建立小型快速回归集和 341 项章节审计集；
- 每个词典适配器至少 50 个分层样本，另保留 30 个结构压力样本；
- 版本对比报告必须列出新增、消失、范围变化、分类变化和解释变化；
- 抽取 5% 自动命中做盲审，避免规则作者只验证自己预期的正例；
- 对误报高频族实施 ablation：关闭某证据后比较精确率、召回率和边界变化，确认权重不是偶然拟合。

完成定义：覆盖报告有稳定分母，主要体裁达到目标，严重错误率达标，且提升不以越界、错误合并或不可解释的首项选择为代价。

## 7. M2：本地文档库、正式导入入口与发布资源边界

### 7.1 规模与目标

- 规模：L，5～7 个提交。
- 目标：用户可以导入、保存、重新打开并继续阅读本地文本；发布版在没有研究词典和网络时仍能完成基础阅读。
- 正式入口：粘贴文本、TXT、Markdown、SRT、基础文档库、章节和阅读位置。
- 已有入口：EPUB 由本地 Rust 适配器转换为 Markdown 后进入现有文档会话。
- 非目标：网页抓取、OCR 和跨设备同步不进入本模块；它们以后通过外部适配器转换为规范文档。

### 7.2 设计

- 新增 `documents`、`document_revisions`、`chapters`、`reading_positions`、`source_imports` 和可选 `bookmarks`。
- 文档身份由规范化原文哈希、来源标识和 revision 组成；章节和位置使用稳定字符锚点，不绑定易变化的前端段落序号。
- 导入器只负责解析源格式和生成规范文档，不直接运行语言分析。
- `DocumentSession` 继续作为打开后的运行态；文档库保存源、元数据、位置和 revision，不复制完整运行 Token 图。
- 保存原始来源文件信息、导入时间、标题、章节、字幕时间轴和编码诊断。
- 建立 `release` 与 `insider` 两套资源清单。公开发布默认只包含许可明确的资源；研究词典必须由显式渠道配置控制。
- 如果公开基础词典不足，优先提供用户导入／重建流程和清晰诊断，不用研究资源填补发布缺口。

### 7.3 验证与实验

- UTF-8、日文 Windows 常见换行、BOM、长行、ruby 标记和 SRT 时间轴 round-trip；规范文本字符重建 100%；
- 10 本 3～8 万字文本连续导入、关闭、重启和恢复，位置误差为 0 个规范字符；
- 导入同一文件去重，源文件变化产生新 revision，不静默覆盖旧标注；
- 进程异常退出后文档与位置恢复；损坏数据库只影响单一文档或可回退快照；
- 打包检查枚举实际资源，`release` 包中不得出现未批准研究词典、原始 MDX、SQLite 缓存或研究文本；
- 首本实验：新用户从安装到导入第一章并开始阅读的中位步骤不超过 3 个主要动作。

完成定义：不依赖当前 textarea 会话也能建立可恢复的本地书库，发布包的资源来源和许可边界可审计。

## 8. M3：分析器选型实验与条件性句法／基础读解

### 8.1 规模与目标

- 规模：先做 M 级选型实验；只有实验通过后再进入 XL 级正式实现，总计 7～10 个提交。
- 目标：先判断现有文节规则、GiNZA、KWJA 或不引入外部分析器哪一方案值得采用；通过后才在不改写规范 Token 的前提下提供句子骨架、从句、引用、接续和局部篇章关系。
- 非目标：完整主语恢复、全篇指代唯一解析、深层语义角色和古典日语不进入 v1.0。

### 8.2 覆盖

- 句内：谓词核心、格成分、连体修饰、状语、并列、条件、原因、让步、引用、补助用言、小句边界；
- 段落局部：接续词功能、指示词候选、显式话题延续、引用来源候选、省略论元候选；
- UI：紧凑句子骨架、当前文节在句中的作用、接续／引用提示和“不确定候选”；
- Provider：统一 `SyntaxAnalysisProvider`，至少有无外部组件的本地 fallback 和 roadmap 指定 sidecar 的验证实现；所有输出映射回规范字符范围。

选型顺序：

1. 现有 Vibrato/IPADIC＋本地文节候选作为基线；
2. GiNZA 验证文节、主辞、依存、谓词和直接论元；
3. KWJA 只抽样验证更深的述语项结构、照应、共指和篇章能力；
4. 分别得出“不引入、可选 provider、正式依赖”的结论，不要求统一选择一个工具承担全部层级。

### 8.3 设计

- provider 输出独立 artifact，不直接改写 Morpheme、Bunsetsu 或 GrammarOccurrence；
- 对齐层保存 provider token、规范 morpheme、字符范围和失败原因；无法对齐时降级，不做猜测式偏移；
- 读解现象通过 M0 协议进入统一 occurrence，解释层只消费结构证据和已识别语法；
- 句子骨架先显示谓词、关键格成分和从句关系，不默认展示完整依存图；
- 指代和省略只返回候选、证据和置信度，不以单一事实显示。

### 8.4 验证与实验

- 从 M0 金标中抽取至少 500 句句法／读解子集，按体裁报告谓词、格成分、小句、引用、接续和范围指标；
- provider 对齐成功率、失败原因和耗时独立报告；对齐失败不得破坏原分析；
- fallback 与 sidecar 结果做差分，确认新增能力和新增错误分别来自何处；
- 结构提示关闭时阅读器结果与当前主线完全一致；
- 先进行 8～12 人可用性试验，再进行不少于 20 人的对照阅读：比较同等文本下的理解正确率、查词／提问次数、返回正文时间和疲劳度；
- 句子骨架只有在理解正确率提升且中断成本没有显著恶化时才默认启用。

完成定义分两级：选型实验完成时，必须明确每个候选是否引入及其使用边界；正式模块完成时，用户能够在正文内理解“本句主干是什么、当前成分起什么作用、前后如何连接”，且错误结构不会反向污染规范文本分析。

## 9. M4：语义学习事件、知识聚合、可重算画像与完整导出

### 9.1 规模与目标

- 规模：XL，6～9 个提交。
- 目标：把当前曝光计数和离散选择升级为可审计、可重放、可迁移的本地学习事实层，为卡片和动态辅助提供唯一状态来源。
- 非目标：本模块不决定复习时间，也不自动判断用户理解成功。

### 9.2 覆盖

事件至少包括：

- 文档打开／关闭、章节开始／完成、位置变化；
- 词典查看、语法查看、解释固定、AI 请求；
- 标记已知／未知、收藏、取消收藏、笔记；
- 分词、词典、语法 sense、表达类型和范围纠正；
- 正式复习结果和自然复现观察；
- 模式切换和辅助显隐，但自然曝光不得直接记为成功复习。

知识项至少覆盖：词汇、义项候选、汉字读音、语法 concept/sense、表达和整句。文本实例保存文档 revision、字符范围、上下文、分析版本和来源。

### 9.3 设计

- `learning_events` 使用 append-only 语义事件，事件带 schema version、occurred_at、document/revision、actor、source 和 payload；
- `review_logs` 保存正式回忆结果，与自然阅读事件分离；
- 派生画像、统计和知识项状态可以从事件重建，允许算法升级后重算；
- 用户纠正具有作用域和撤销事件，不直接覆写历史事实；
- 现有 exposure、known、segmentation 和 dictionary choice 表通过一次性迁移生成等价事件或明确保留为兼容投影；
- 完整导出包含文档、知识项、文本实例、事件、卡片状态、来源和版本；提供 JSON 主格式与 CSV 视图；
- 删除支持按文档、时间范围和全部本地数据执行，并验证派生状态同步清理。

### 9.4 验证与实验

- 同一事件流重复重放，派生画像字节级或规范 JSON 等价；
- 乱序、重复、撤销和旧 schema 迁移有固定 fixture；
- 文档删除后不存在孤立实例、卡片引用或缓存泄漏；
- 10 万、100 万事件规模下测量写入、重放、聚合和导出耗时；阅读热路径采用批量或异步写入；
- 导出后在空数据库重新导入，知识项、实例、事件和卡片状态 round-trip；
- 隐私审计确认正文和词典内容不会因事件记录被意外发送或写入日志。

完成定义：任何用户状态都能回答“由哪些事实得出”，算法变化后可以重算，用户可以完整导出和删除。

## 10. M5：动态卡片、复习队列与 FSRS

### 10.1 规模与目标

- 规模：L～XL，5～8 个提交。
- 目标：从知识项和文本实例动态投影卡片，提供轻量复习闭环，并让正式复习结果反哺用户状态。
- 非目标：不建设公开卡片社区，不要求用户手工维护传统牌组，不把每次自然遇见记为正式复习。

### 10.2 覆盖

- 卡片类型：词汇、义项、语法／表达、整句；
- 生命周期：candidate、active、suspended、retired；
- 复习状态：new、learning、review、relearning；
- 内容：规范知识项、当前最合适的私人例句、出处、必要提示、答案和来源；
- 队列：到期复习、本书高频反复卡顿、用户显式收藏；
- 导出：卡片 JSON/CSV，保留知识项和事件关联。

### 10.3 设计

- 卡片是知识项、实例和用户策略的投影，不作为底层知识事实；
- FSRS 状态独立保存稳定性、难度、到期时间和最近正式复习，不从曝光次数伪造；
- 私人例句可随更优上下文更新，但保留历史来源；
- 同一知识项跨文档聚合，义项不确定时不静默合并成一个确定卡片；
- 复习结果写入 M4 `review_logs`，再由投影更新调度状态。

### 10.4 验证与实验

- 使用固定时间线验证 FSRS 状态转移、时区、撤销和导入导出；
- 随机事件流与参考实现或离线计算结果比较；
- 同一知识项多实例、义项分裂／合并和例句替换有回归 fixture；
- 复习页面在 100、1,000、10,000 卡片规模下保持稳定分页和查询；
- 先做 14 天内部试验，记录完成率、单次耗时、7 天回忆率和放弃原因；
- 再比较“纯到期队列”与“到期＋本书反复卡顿加权”，只在后者提高回忆且没有显著增加疲劳时进入默认策略。

完成定义：用户读到的内容能自动形成可解释卡片，正式复习可持续运行，调度状态不依赖不可审计的曝光启发式。

## 11. M6：自适应阅读辅助、阅读模式与密度预算

### 11.1 规模与目标

- 规模：L，5～7 个提交。
- 目标：在不改变分析事实的前提下，根据用户显式模式、知识状态、遗忘风险和当前文本负担控制提示强度。
- 原型来源：可以审阅 `backup-master` 的 `60bbf0d`，但不能直接把该原型视为完成状态或无条件 cherry-pick。

### 11.2 覆盖

稳定 `SessionMode`：

- `immersion`：最低常驻提示和最高中断成本；
- `balanced`：默认低干扰辅助；
- `textbook`：强调目标现象、讲解和缺口；
- `intensive`：最高结构提示，但正式复习仍需用户显式进入。

干预等级：

```text
0 无常驻提示
1 低对比度标记
2 悬浮／点击时优先显示
3 紧凑提示或自然复现
4 用户主动展开完整讲解／复习
```

### 11.3 设计

- 模式是用户选择，自动策略只能在模式内部调整，不静默切换模式；
- `InterventionScore` 至少消费知识状态、遗忘风险、文本重要性、置信度、近期重复和当前密度；
- 密度预算按段落或视口限制同时出现的词汇、语法、表达和读解提示；
- 分析 occurrence 不因模式变化重新计算，只改变 presentation projection；
- 模式切换、提示展示和用户主动打开写入 M4 事件；
- 自然复现只能作为观察信号，不能直接生成正式复习成功。

### 11.4 验证与实验

- 给定固定用户状态、文档和模式，投影结果完全确定；
- 切换模式不触发 NLP 重算、不改变滚动位置、不关闭当前气泡；
- 密度预算在窄屏、墨水屏和长段落下不溢出；
- 使用事件回放比较策略版本，报告提示数、误隐藏、主动查询、长停顿和章节完成；
- 先做 4 模式可用性测试，再对 `balanced` 与自适应 `balanced` 做交叉实验；
- 默认策略必须同时满足：理解度不下降、每千字无效提示减少、返回正文时间不恶化、用户疲劳度不增加。

完成定义：系统能够减少已知内容干扰，优先暴露真正阻碍，但不会以自动打断或错误隐藏牺牲连续阅读。

## 12. M7：BYOK AI 上下文问答、授权与降级

### 12.1 规模与目标

- 规模：L，5～7 个提交。
- 目标：在本地分析不足时，用户可以对当前词、句、选区或读解结构主动请求提示、解释或译文；AI 不可用时基础阅读完全可用。
- 非目标：不默认上传全文，不自动采用分词／词典／语法修改，不让模型回答成为规范事实。

### 12.2 覆盖

- 任务：词典候选解释、语法／表达解释、句子骨架补充、指代／省略候选、提示、详细解释、手动译文；
- Provider：通用 BYOK 接口、模型配置、能力发现、取消和超时；
- 授权：每次请求显示将发送的文本范围、结构证据、词典片段和预计大小；
- 输出：引用原文片段和本地证据，区分确定、推测、信息不足；
- 采用：模型只能给建议，用户操作后才生成纠正或选择事件。

### 12.3 设计

- 扩展现有 dictionary assistant schema 为通用 `AssistantTaskRequest/Response`，保留任务类型和证据预算；
- core 只定义纯数据和强校验，网络、凭据、重试、取消和速率限制由 Tauri 宿主实现；
- 凭据使用系统安全存储或等价机制，不进入日志、导出和前端持久明文；
- 请求上下文按最小片段构建，默认不发送整章、历史数据库或未选中的词典正文；
- 回答缓存键包含任务、模型、prompt 版本、规范片段哈希和证据版本；
- 离线、超时、拒绝授权和输出无效都回退到本地解释，不阻断气泡和阅读器。

### 12.4 验证与实验

- 使用 mock transport 覆盖成功、取消、超时、无效 JSON、prompt injection、超预算和 provider 错误；
- 本地强校验拒绝模型新增不存在的候选、越界引用、未授权采用和伪造高置信度；
- 200 个分层问题集分别评估词典、语法、句法、指代和译文，报告正确、部分正确、误导、信息不足和引用错误；
- 记录每类请求的发送字符数、词典字节数、延迟和费用估算；
- 进行人工盲评，比较“本地结果”“本地＋AI 提示”“本地＋AI 解释”的正确率、解决时间和误导率；
- 只有误导率达到发布阈值、授权路径清晰且离线降级完整时才默认显示入口；否则保留实验开关。

完成定义：AI 是显式、最小上下文、可取消、可审计的增强层，失败或关闭时 v1.0 的本地阅读闭环仍完整。

## 13. 发布门与最终验收

### 13.1 机械正确性

- 规范文本重建：100%；
- 字符范围完整性：100%；
- 硬边界越界：0；
- 删除规则、撤销选择和关闭 provider 后能够恢复规范结果；
- 冷启动、渐进、暖缓存和重放的最终稳定 artifact 等价；
- schema、目录、事件和导出不存在悬空引用。

### 13.2 语言质量

- 现代标准日语目标目录或现象召回达到 90%～95%；
- accepted 高精确率，严重误导性错误低于 5%；
- 指标按体裁、category、来源和置信度分层；
- pending／unknown 有明确原因和后续队列。

### 13.3 产品闭环

至少完成以下端到端场景：

1. 导入一本 3～8 万字现代文本；
2. 关闭并恢复章节位置；
3. 阅读中查词、查看语法／读解、进行纠正或主动 AI 提问；
4. 自动形成知识项和文本实例；
5. 收藏或自动生成动态卡片；
6. 完成正式复习并写入 review log；
7. 下一章提示根据状态变化；
8. 完整导出、删除并验证离线继续使用。

### 13.4 性能、资源与发布

- 保持现有长文档首批、渐进和暖缓存基线，不因事件、句法或卡片引入同步阻塞；
- 100 万事件、10,000 卡片和多本文档下仍可重放、查询、导出；
- `release` 与 `insider` 资源清单自动检查；
- 公开包不包含研究文本、原始 MDX、生成 SQLite 或未批准词典；
- `cargo test -p kotoclip-core`、`cargo check -p tauri-app`、`npm run test:ui`、`npm run build` 和模块专项审计通过。

### 13.5 真实使用实验

发布前至少进行两层实验：

- 内部纵向：8～12 名目标用户完成 14 天、至少 2 章阅读和一次复习；
- 外部阅读测试：不少于 20 名约 N3～N2 用户，比较当前版本与 v1 候选版的章节完成、主动查询、理解正确率、返回正文时间、疲劳度和 7 天回忆。

v1.0 的核心成功信号不是提示数量增加，而是用户更稳定地读完章节、减少无效中断，并在 7 天后保留可测量的知识收益。

## 14. 建议实施顺序

### 14.1 第一批：冻结发布事实

1. M0 统一协议与评测；
2. M2 文档库和发布资源；
3. M1 第一轮目录分母、N1/N2 来源与 341 项表达审计。

完成后应具备技术预览版的稳定发布基础。

### 14.2 第二批：形成阅读测试闭环

1. M3 句法与基础读解；
2. M4 事件、知识聚合和完整导出；
3. M1 持续按体裁扩充与误报收口。

完成后可以开始真实阅读测试，不应等待卡片系统全部完成才收集行为数据。

### 14.3 第三批：完成 v1.0 学习闭环

1. M5 动态卡片与 FSRS；
2. M6 自适应阅读辅助；
3. M7 BYOK AI 问答；
4. 全量语言质量、性能、许可、导入导出和 14 天实验验收。

## 15. 状态记录格式

每个模块完成一个提交组后，在该模块权威文档追加：

```text
日期：
提交：
范围：
协议／schema：
代码与数据规模：
正确性结果：
覆盖与质量结果：
性能与资源结果：
用户实验结果：
遗留项：
下一依赖：
```

状态词统一使用：

- `未开始`：没有正式 schema 或代码；
- `框架完成`：协议、入口和最小闭环存在，但没有发布覆盖；
- `主要实现完成`：核心架构稳定，没有 P0 阻塞，仍有样本驱动扩展；
- `发布验收中`：功能完整，正在完成覆盖、性能、许可或用户实验；
- `完成`：达到本文件和模块权威文档的全部完成定义。

不得仅因存在 UI、提交标题或一次高覆盖数字而标记“完成”。
