## 2. 已确认的系统性问题

### 2.1 查询、候选与排序

1. **读音命中被当成词义相关性。** `is_preferred` 主要表示读音相同，不表示表记、词性、词汇类别或上下文语义相符。`その`、`もう` 等纯假名词因此把大量同音汉字词标成“首选”。
2. **缺少词性约束。** 阅读器已经拥有正文词性，但查询结果没有用它区分连体词、接续词、副词、普通名词、姓氏、汉字音义条目和接辞条目。`その` 的连体词被 `園` 压过；`ただし` 混入古语形容词 `正し`。
3. **缺少词条类型约束。** 大辞林 `p[type="漢字"]` 的汉字音义条目与普通词汇条目并列返回，`前`、`もう` 等词因此出现大量并非正文词义的“漢／音”条目。
4. **表记变体没有强弱等级。** `前` 可作为 `先` 的异表记，结果仅表记查询时 `先［さき］` 反而排第一；罕见、历史、仅构词使用的表记与现代常用表记没有区分。
5. **接辞读音被当作独立读音。** 小学馆的 `もう-`、`-もう` 等连字符表示前接或后接成分，却进入普通读音匹配；`猛［もう-］` 被标为 `もう` 的首选。
6. **サ变词读音模型不完整。** `反響する` 的词头包含 `する`，但结构化 reading 只有 `はんきょう`，与正文 `はんきょうする` 不匹配，导致唯一正确词条也不是 preferred。
7. **外来语词头污染 reading。** `シルエット〖silhouette〗` 的 reading 变成整个带外文注记的词头，而不是 `シルエット`。
8. **无上下文查询默认顺序危险。** `気配` 默认先给 `きはい`，`人間` 默认先给 `じんかん`，`前` 默认先给 `先［さき］`，与现代常见用法明显不符。
9. **同读音下全部 preferred。** `その` 的 `園（姓氏）`、`其の`、`園（庭园）` 都是 `その`，带读音查询后全部 preferred，preferred 标识失去选择能力。
10. **缺少查询证据说明。** `いつの間に` 仅靠 reading fallback 命中 `いつのまに`，UI 不会说明“表记未精确命中、仅由读音找到”。
11. **无结果与错误候选之间没有质量门。** `もう` 没有返回目标副词，但仍返回 15 条同音或异读汉字词，用户看到的是大量错误内容而不是明确的“未找到合适词条”。
12. **同一词典中的多个 entry 被统一视为同层结果。** 姓氏条、普通名词条、汉字条、古语条、重定向条均没有 entry kind，无法正确排序或折叠。

### 2.2 源包构建与记录边界

1. **小学馆同表记多读音记录被拼接。** `毛` 的原始定义同时包含 `毛［け］` 与 `毛［もう］` 两套 `<link><h3><section>`；`網` 同时包含 `網［あみ］` 与 `網［-もう］`。这不是一个词条的自然多义，而是多个源记录被合并到一个 definition。
2. **适配器只取第一个小学馆读音。** `SHOGAKUKAN_READING_RE` 只读取首个 `pinyin_h`，后续拼接记录的读音和身份丢失；`毛［もう］` 内容被挂在 `毛［け］` 下，`網［-もう］` 被挂在 `網［あみ］` 下。
3. **同表记多读音需要在构建期保留独立 entry。** 当前按 headword 聚合的边界疑似过粗，至少对小学馆不成立。
4. **表记中的词尾结构未提取。** Crown 的 `mj_katsuyogobi` 明确把 `する` 标为活用语尾，但构建和 presentation 没有把它转换成 headword stem、suffix、完整 display reading。
5. **外语词源仍混在词头字符串。** `シルエット〖silhouette〗` 未拆成现代词头、来源语言、原文拼写和词源说明。
6. **历史假名、现代读音、音调数字未拆分。** `まへ`、`かはいい`、`ゑん`、`まう` 与现代读音仍作为 `.ruby` 或普通文本混在正文。
7. **构词专用读音缺少 bound-morpheme 类型。** `もう-`、`-もう` 应进入接头/接尾或音读构词信息，不能与独立词 reading 共用字段。

### 2.3 HTML 适配与结构化管线

1. **当前仍基本是“正则删头 + sanitizer + rich_text”。** 三本词典最终都主要输出一个 `rich_text`，没有真正拆出词头、读音、词性、义项、子义项、例句、译文、标签、用法、词源、派生、惯用句等结构。
2. **大辞林链接删除会破坏正文标点。** 例如：
   - `うける` 的 `生ずる。⇔与える。` 清洗后变成 `生ずる。。`。
   - `前` 清洗后出现 `場所。。【・】。`、`直前の。。。`。
   - `先` 多处反义链接移除后留下 `。。`。
3. **嵌套链接删除留下不可读残片。** `前` 原文中的 `しりえ【後方・後】` 含嵌套链接，删除 anchor 后留下 `【・】`。
4. **小学馆引用删除留下裸箭头。** `其の`、`猛` 等清洗后保留单独的 `⇒` meaning 行，既无目标又被当成正文。
5. **链接关系失去所属义项。** 大辞林关系被提升到 entry 全局，无法知道某个反义词、参照词属于哪个 sense 或 subsense。
6. **子项目/句项目分类失败。** `人間` 的 24 个子项目和 4 个句项目都输出为 `related`；原因疑似 `classify_relation` 从最后一个 `<br>` 之后取上下文，已经跳过 `〈子項目〉/〈句項目〉` 标记。
7. **去重过早且过粗。** 同一 target 在不同义项出现会被全 entry 去重，可能丢失“多个义项各自有同一反义词”的语义位置。
8. **sanitizer 会重排非法旧式 HTML。** `深い` 的净化 DOM 中，`leftnull` 只包住第 ① 项，②–⑩ 被移到外层；层级破坏不是 CSS 可修复问题。
9. **依赖被 sanitizer 删除的 inline style。** 大辞林 ㋐–㋖ 原本使用 `style="float:left"`，清洗后 style 被移除，却没有补充结构类，子义项编号和正文失去对齐依据。
10. **清洗后产生空容器。** 大量词条尾部出现一个或多个空 `<p></p>`，增加无意义间距。
11. **伪 ruby 未转换为真正 ruby。** 大辞林 `.ruby` 只是跟在汉字后的缩小文本，如 `門かど`、`お祓はらい`、`関かかわり`；Crown 的 `ruby_box/mj_rb/rt_box` 也只是 `真（ま）` 的内联结构。
12. **词头重复策略不一致。** Crown 和小学馆删掉原词头，Daijirin 保留 `bss + hy`，导致 UI 顶部已有词头时，大辞林正文再次显示读音、音调和表记。
13. **省略符号未展开。** 大辞林例句中的 `━`、`—・る`、`—・い`、`━ げ` 等仍要求读者自行还原词头和活用。
14. **多条例句仍在一个 span 中。** 大辞林 `.rei` 常包含多组引号，无法单独换行、折叠、复制或关联义项。
15. **标签没有统一语义。** `文、可能、派生、慣用、表記、漢、音、成語、補足、注意、ことわざ` 分散在 `.deco`、`.white-square`、`.black-square`、`type` 和普通文本中。
16. **Crown 专用结构覆盖不全。** `group_kanyo`、`group_kotowaza`、`mean_sakuhin_jp/cn`、`mean_hosetsu`、`mean_iikae`、`mean_eiyaku_kubun` 等保留在 HTML，但 CSS 未全面适配。
17. **小学馆 `level/no/type` 已保留却未转换。** 这些字段足以构建 sense tree，目前仍只靠 CSS 选择器展示。
18. **内容块协议名义结构化、实际单块。** `content_blocks` 基本等同 `definition_html` 的复制，无法支持模块级折叠、去重、双语布局和偏好控制。

### 2.4 UI 编排与交互

1. **“释义 N”判断错误。** `TooltipPanel.vue` 只要当前词典当前读音下有多个 entry，就标为“释义 1/2…”。实际可能是：
   - `受ける` 与 `請ける` 的不同表记；
   - `園` 姓氏、`其の` 连体词、`園` 普通名词；
   - 普通词条与 `p[type="漢字"]` 汉字条；
   - `但し` 与古语 `正し`；
   - 多个完全不同汉字的同音词。
2. **entry headword 与 entry kind 没有分工。** UI 只能显示粗粒度 headword，不能标示“姓氏”“汉字音义”“接辞”“古语”“重定向”“异表记”。
3. **读音选择不足以解决纯假名歧义。** `その` 的多个错误词条读音完全相同，切换 reading 没有帮助。
4. **词典切换可能放大错误默认值。** 例如 `その` 的 Crown 只有 `園`，若默认优先 Crown，用户首先看到的是“花园”而非连体词。
5. **多词典互补不可见。** `楽しむ` 在 Crown 只有粗略译语和两个例句，大辞林有五义；切换后信息量骤变，没有“本词典覆盖较少/另一本有更多义项”的提示。
6. **正文身份与词典身份可能冲突。** 表头可能显示正文 `その`，正文区却显示 `園`；用户不易判断当前内容为什么出现。
7. **重定向空正文仍会显示。** `正し` 清洗后只剩词头和词性，但 `content_blocks` 非空，因此仍作为完整 entry 卡片显示。
8. **关系列表与 sense 脱离。** 反义、参照、子项目统一放在 entry 底部，无法在阅读定义时理解关系对应位置。
9. **大量关系只能折叠成无类别列表。** `人間` 28 个关系几乎全标 `related`，折叠虽避免撑高，但导航质量很低。
10. **CLI 预览与真实 UI 并不等价。** CLI 只接收 `Vec<DictEntry>`，没有完整 `DictionaryLookup`，无法复现 candidates、selected_target、dictionary_names、不可用状态和真实词典选择。
11. **CLI 同时展示多个词典卡片，真实 UI 只展示活动词典。** 因而 CLI 不能直接验证默认活动词典是否错误。
12. **CLI 无法观察纯假名表记候选。** `その/もう/うける` 的关键问题恰好依赖 candidates 与 selected_target，当前研究工具缺失最重要状态。

### 2.5 CSS、字体与排版

1. **整个气泡继承日文字体。** 中文释义和中文例句没有 `lang=zh-CN` 或中文字体栈，容易使用日文字形。
2. **Crown 拼音和英语占据大量视觉空间。** 拼音紧跟每个中文词和每条例句，英语又以内联小块重复提供，正文中心被稀释。
3. **小学馆日中例句没有分行。** `jae` 与 `ja_cn` 都是 inline，只用 8px 左边距，长句在气泡中混成一行或随机换行。
4. **Crown 日中例句也没有可靠分行。** 日文、斜杠、中文、拼音处于同一行内流。
5. **中文标点与日文/ASCII 标点混用。** 可见 `，、。．, .；;?!` 同时存在；Crown 例句频繁使用 ASCII comma，拼音也带原始标点。
6. **特殊空格未规范化。** ` `、`&nbsp;`、全角空格等用于读音分节和音调，直接进入 UI。
7. **大辞林不同 hierarchy badge 视觉相同。** `.deco` 不区分 `rect/invert-rect/round-rect` 的真实层级语义；`随分` 的顶层“一/二/三”和下层“一/二”容易混淆。
8. **`.black-square` 未适配。** 小学馆 `随分` 的一级编号使用 black-square，但现有 CSS 只覆盖 white-square。
9. **Crown 惯用句和谚语缺少专用样式。** `可愛い` 的 proverb、`うける/反響する` 的惯用句容易与正文黏连。
10. **Crown 作品名、作者、换言结构缺少样式。** `園` 的《樱桃园》、契诃夫信息以及 `ごちゃごちゃにする［なる］` 没有清晰语义层次。
11. **例句缩写下划线过密。** Crown 每个例句中的词头都使用 dotted underline，长条目中形成持续视觉噪声。
12. **小学馆 meaning 使用 flex 过于粗暴。** 数字、语义区分、多个中文译词、标签都在同一 flex 行内，换行位置不可控。
13. **小学馆所有 `<b>` 同时承担编号与中文关键词。** 当前统一 accent 处理会让编号和多个译词都抢占视觉中心。
14. **补充说明没有统一样式。** CSS 仅特判 `type="注意"`，`補足` 常按普通义项显示。
15. **伪 ruby 只缩小、不上标。** 直接导致用户已经指出的“汉字后跟小字”问题。
16. **窄宽气泡中的长例句缺少双语网格或块布局。** 460px 左右宽度下，中日文和拼音混排极易形成难读的长段。

## 3. 按词顺序记录的问题

### 3.1 差し掛る

1. 只命中大辞林；`差し掛る` 与更常见的 `差し掛かる` 是否存在跨词典表记变体未建立，可能造成小学馆/Crown 漏查，待核验源库。
2. UI 表头可能显示 `差し掛る【さしかかる】`，正文又显示 `さし かか・る【差し掛（か）る】`，表记和读音重复且写法不一致。
3. `差し掛（か）る` 的括号内送假名没有结构化为可选表记或规范写法。
4. `4/0` 音调数字没有标签，用户无法理解多个数字代表什么。
5. `(動ラ五)` 与 UI 已有词性可能重复；辞典缩写也未转成统一标签。
6. 三个 sense 仍是圈号 HTML，没有统一 sense model。
7. 例句 `—・る` 未还原完整词形。
8. sanitizer 产生尾部空 `<p>`。

### 3.2 気配

1. 仅表记查询把 `きはい` 排在现代常用 `けはい` 之前。
2. `けはい` 同时有 Crown 和大辞林，`きはい` 又是另一大辞林 entry；只有上下文读音才能暂时纠正排序。
3. Crown 译词包含 9 组拼音和英语 `a sign/an indication`，视觉冗余明显。
4. Crown 六条例句日文、中文、拼音同一行内流。
5. 中文继承日文字体。
6. 大辞林把读音、音调、表记和词源说明再次放入正文，和气泡表头重复。
7. `けはい` 的词源/表记历史说明应独立为 orthography note，不应挤在首行。
8. `きはい` 例句中的 `門<span class="ruby">かど</span>` 显示为汉字后小字，不是真 ruby。
9. 大辞林一个 `.rei` 内包含多条例句，无法分行。
10. `きはい → けはい` 的关系被统一标为 reference，实际上更接近现代读法参照或推荐读法，关系粒度不足。
11. `気配交換` 从正文移到关系区后失去所属 sense。

### 3.3 うける

1. 同一读音返回 Crown `受ける`、大辞林合并表记条、小学馆 `受ける`、小学馆 `請ける`，前端会把小学馆两条误标为“释义 1/2”。
2. 大辞林一个 entry 内同时包含 `受・請・承・享`，小学馆却拆为独立 headword；多词典没有统一表记模型。
3. Crown 七义、大辞林十二义、小学馆七义加独立 `請ける`，切换词典时层级和覆盖范围剧烈变化。
4. Crown 第 7 义只有日文区分和英语 `take`，没有明确中文中心译义。
5. Crown `真（ま）` 是伪 ruby 结构，CSS 未转成真正 ruby。
6. 大辞林 ④ 下有 ㋐–㋓，inline float 被清洗掉后子义项布局失效。
7. 大辞林删除 `⇔与える` 后留下 `生ずる。。`。
8. 大辞林 `お祓はらい`、`質種しちぐさ` 等均为错误的后置小字。
9. 大辞林 `慣用/表記` 仍是正文内联片段，不是独立模块。
10. `表記` 说明又重复大量前文例句，未进行信息去重或折叠。
11. 例句中的 `—・ける/━` 未展开。
12. 小学馆 42 条双语例句全部日中同行，长词条极难扫读。
13. 小学馆 `[補足]` 没有专用样式。
14. 小学馆可见 `┏` 等排版残片。
15. Crown/小学馆均保留大量拼音或中日混排问题。

## 5. 当前已经可以确定的改进边界

1. **不能继续以 CSS 修补为主。** `深い` 的 DOM 已被 sanitizer 改写，`前` 的正文已被正则删坏，必须在 sanitizer 前完成词典专用解析。
2. **必须引入词典中间结构。** 至少需要 entry identity、form/readings、entry kind、POS sections、sense tree、examples、translations、labels、notes、relations、subentries、etymology、pitch 信息。
3. **查询和呈现必须解耦。** 查询负责“为什么命中、是否适合正文”，适配器负责“源结构是什么”，UI 负责“如何显示”；不能再用 reading 相同直接代表优先结果。
4. **正文查词和主动搜索必须分流。** 正文查词应严格使用表记、词性、形态和上下文证据；主动搜索才可容许宽泛同音候选。
5. **每本词典需要独立 adapter。** 通用层只能消费统一结构，不能继续让 Vue/CSS 理解原始 class。
6. **中文内容必须带语言属性和中文字体栈。** 日文、中文、拼音、英语应有独立 typography role。
7. **双语例句必须成为成对结构。** 默认日文一行、中文下一行；拼音默认隐藏或不生成。
8. **表头必须单一来源。** 词头、读音、表记、外文来源和音调应统一在 header/choice controls；正文 adapter 不应重复输出原始 midashi。
9. **“释义 N”必须基于 sense/entry kind，而不是 entries.length。** 不同表记、姓氏、汉字条、古语条、redirect 不得伪装成同一词条的多个释义。
10. **研究 CLI 必须升级为完整 lookup 预览。** 后续验收需要同时固化完整 JSON 状态和与真实 TooltipPanel 等价的单活动词典视图。
