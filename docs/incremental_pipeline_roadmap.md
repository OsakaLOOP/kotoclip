# 加载、增量分析与首屏性能重构追踪

## 1. 目标

本任务统一解决三个互相依赖的问题：

1. 将当前整篇、顺序、一次性产出 `Vec<AnnotatedToken>` 的分析流程重构为可拆分、可复用、可审计的阶段管线。
2. 建立正式的文档会话、阶段失效与增量更新协议，替换现有未接入且需要整篇 Token 往返的局部刷新补丁。
3. 使虚拟 DOM 与后端按范围分析真正一致：首屏只阻断必要阶段，其余工作按视口优先级在首帧后完成。

最终目标不是单点缩短某个函数，而是同时降低冷加载时间、规则更新延迟、局部交互延迟、IPC 体积和无效重算量。

## 2. 固定约束

- 保持原文字符可逆、字符范围完整、规则结果稳定和曝光不重复记录。
- 不改变既有正式执行语义：

  ```text
  Morpheme → WordFormation → DictionaryLexical → Bunsetsu → Grammar
           → Profile → Expression → Presentation
  ```

- 构词、词典词汇、文节、语法、画像和表达必须保持独立阶段，不能为了局部刷新重新混成一种规则或注解。
- 只有构词、词典词汇和文节阶段可以改变正式 Token 边界；语法、画像和表达只能在既有边界上追加或更新注解。
- 词典字符串滑窗没有结构化证据时继续保持 pending，不得因性能重构提升为正式惯用语。
- 现有规则 schema、候选接受/拒绝语义、稳定词条键和 N-best 原始证据必须保留。
- 增量管线以后端规范状态为准，前端不得通过上传整篇可变 Token 维持后端状态。

## 3. 基准与验收口径

### 3.1 当前基准

当前第一话测试存在两组相近但输入统计略有差异的结果：

- 用户实测：46,301 字，6,618 ms。
- 2026-07-13 本轮 CLI `reader-benchmark`：原始 48,597 字、实际分析 44,661 字，端到端 6,249 ms；其中 Engine 初始化 541 ms、分析 5,298 ms、紧凑 IPC 序列化 395 ms、负载 4,445,466 bytes。

当前细分计时中，“文节与振假名”占 4,491 ms，但该名称实际混合构词、词典词汇候选、SQLite 查询、文节 DAG 和 ruby 合并，不能直接作为单一算法结论。

### 3.2 统一回测要求

每次性能提交必须记录：

- 源文件、章节标题、原始字符数和实际分析字符数；
- 冷启动初始化、核心分析、IPC 序列化和端到端耗时；
- 拆分后的各阶段耗时；
- Token 数和 IPC 字节数；
- 进程峰值工作集或等价内存指标；
- 同一命令至少执行三次，分别记录首次和后续结果，不只报告最优值。

性能对比必须使用同一输入、同一 profile 副本、关闭真实曝光写入，并区分编译时间与运行时间。

## 4. 总体架构

### 4.1 阶段产物

每个阶段输出显式 artifact，不再只保留最终 Token：

```text
PreparedDocument
└─ SegmentArtifact[]
   ├─ MorphemeArtifact
   ├─ WordFormationArtifact
   ├─ DictionaryLexicalArtifact
   ├─ BunsetsuArtifact
   ├─ GrammarArtifact
   ├─ ProfileArtifact
   └─ ExpressionArtifact
```

阶段必须声明输入版本、输出版本、局部计算边界、上下文半径、是否改变 Token 边界和缓存键组成。

### 4.2 文档会话

后端维护 `DocumentSession`：

```text
DocumentSession
├─ session_id / document_id / revision
├─ normalized source / source map
├─ segment index / stable span ids
├─ stage artifacts
├─ rule, dictionary and profile fingerprints
└─ scheduler state
```

前端只发送文本编辑或领域 mutation，后端返回带 revision 的 `AnalysisPatch`。Patch 包含变化范围、删除 ID、追加或替换的 Token/注解及进度，不返回整篇结果。

### 4.3 失效传播

| 变化 | 最早失效阶段 | 典型范围 |
| --- | --- | --- |
| 表达规则增删改 | Expression | 全篇候选扫描，但只更新表达注解 |
| 已知/未知与曝光变化 | Profile | 相同词条键对应 Token |
| N-best 选择 | WordFormation | 当前安全 segment 及其下游 |
| 构词规则变化 | WordFormation | 候选命中 segment 及其下游 |
| 词典词汇规则或词典版本变化 | DictionaryLexical | 受影响查询词所在 segment |
| 文节规则变化 | Bunsetsu | 受影响 segment |
| 语法规则变化 | Grammar | 受影响文节或 segment |
| 局部文本编辑 | Normalize/Morpheme | 扩展到安全边界后的局部范围 |
| 纯 UI 状态 | 无 | 不进入分析管线 |

## 5. 分 P 实施计划

## P1：管线拆分、热路径重构与章节级批处理

状态：已完成（2026-07-13）。

### 范围

- 拆开形态素后处理、构词、词典候选生成、词典批量查询、词典候选解析、文节 DAG、ruby 合并和语法计时。
- 引入段级中间产物，但暂不建立长期 `DocumentSession`。
- 将按 content segment 发生的 SQLite 小批次查询改为章节级查询汇总和分发。
- 正式阅读热路径只生成最终文节；完整边界证据、重建字符串和范围诊断保留在审计入口。
- 修复能够确认的跨阶段重复扫描，不改变规则结果。
- 对候选生成、文节选择和注解覆盖进行针对性复杂度优化。

### 性能目标

- 同一第一话基准端到端进入 3–4 秒区间；若环境抖动，以三次运行的中位数不超过 4,000 ms 为目标。
- 核心分析结果、Token 数、字符重建和代表性规则结果保持一致。
- 峰值内存相对当前基准不出现明显增长；暂定中位峰值工作集增长不超过 10%，如测量误差较大需记录原始数据说明。

### 验证

- 运行 pipeline、词典词汇、文节和代表性表达相关测试。
- 运行 `reader-benchmark` 三次并记录阶段细分。
- 运行章节审计入口确认字符重建和范围完整性。
- 检查 release/开发运行时峰值内存。

### 提交

独立提交，建议标题：`perf: 拆分并批处理分析管线`。

## P2：文档会话、稳定身份与 Patch 协议

状态：待开始，依赖 P1。

### 范围

- 新增 `DocumentSession`、revision、阶段版本指纹和生命周期管理。
- 为 segment、token、expression span 建立稳定 ID。
- 新增 `open_document`、`request_range`、`apply_mutation`、`close_document`。
- 设计紧凑 `AnalysisPatch`，支持字符串表增量扩展。
- 前端 store 改为按 ID 合并 patch，虚拟列表不再整页替换对象。

### 验收

- 首次完整结果与旧 `analyze_text` 等价。
- 乱序或过期 revision 不得覆盖新状态。
- Patch 应用后原文重建、Token 顺序和表达范围保持一致。

## P3：领域 mutation 与正式失效引擎

状态：待开始，依赖 P2。

### 范围

- 建立阶段 DAG、失效原因、上下文扩展和传播规则。
- 表达规则保存/删除只重跑 Expression；移除整页 `triggerAnalysis(false)`。
- 已知/未知、曝光、N-best 和各规则目录更新分别接入最早失效阶段。
- 将现有 `refresh_expression_annotations(Vec<AnnotatedToken>)` 降级为兼容入口并最终删除。
- 表达规则编译为按首原子特征索引的 matcher，减少全起点扫描。

### 验收

- 每种 mutation 记录实际重算阶段和字符范围。
- 表达规则更新不得重跑形态素、构词、词典词汇、文节和画像。
- N-best 选择不得重算无关 segment。
- 曝光只记录一次，刷新不得产生重复写入。

## P4：首屏优先、范围分析与虚拟 DOM 协同

状态：待开始，依赖 P2/P3。

### 范围

- 调度单位改为 segment batch，并支持 P0 首屏、P1 overscan、P2 文档其余区域、P3 空闲任务。
- 首屏阻断到稳定 Token 边界和基本画像；表达注解、曝光写入及非首屏处理以 patch 后置。
- 虚拟列表按缺失 range 请求分析，按滚动方向预取。
- 预留语法 badge 或采用稳定布局策略，避免后置注解造成明显布局跳动。
- 规则库、导出、N-best 和词典正文保持按需加载。

### 验收

- 首个可读帧不等待整篇分析。
- 快速滚动时不得显示错误范围或旧 revision；允许短暂骨架，但应有可测缺页率。
- 首屏 IPC、前端解码和主线程长任务显著下降。

## P5：缓存、持久化与暖启动

状态：待开始，依赖 P2/P3。

### 范围

- 建立 session 内存缓存和可选持久阶段缓存。
- 缓存键包含文本片段 hash、系统词典、外部词典、规则目录、schema 和算法版本。
- 画像和表达等用户态阶段与稳定 NLP 阶段分层缓存。
- 增加缓存容量、LRU、版本淘汰、损坏恢复和空闲期落盘。

### 验收

- 暖启动可以先恢复可读结果，再验证或补算失效阶段。
- 任一版本指纹变化不得复用不兼容 artifact。
- 缓存损坏或缺失时自动退化到正常分析，不影响正确性。

## P6：接口收束、审计与基准固化

状态：待开始，依赖前述各 P。

### 范围

- 删除旧整篇 Token 回传刷新链和已无调用的兼容接口。
- 固化加载、局部 mutation、首屏、滚动和暖启动 benchmark。
- 在开发指标中显示缓存命中率、重算字符比例、Patch 大小和各优先级队列状态。
- 更新 README、架构入口、审计方法和故障排查文档。

### 验收

- 完整加载、局部更新、首屏和暖启动均有稳定可复现的基准。
- 各阶段规则审计与性能验证互不混淆。
- 所有公开入口有明确生命周期、版本和错误恢复语义。

## 6. 首屏阻断边界

| 工作 | 首屏是否阻断 | 原因 |
| --- | --- | --- |
| 文本规范化、字符映射、首屏分段 | 是 | 后续范围与稳定 ID 的基础 |
| 首屏形态素、构词、词典词汇、文节 | 是 | 决定正文 Token 和 DOM 结构 |
| 首屏基本画像 | 是 | 决定胶囊基本呈现 |
| 首屏语法 | 原则上是 | 当前成本较低，badge 可能影响布局 |
| 表达注解 | 否 | 可以在首批 Token 显示后 patch |
| 曝光写入 | 否 | 应在首帧后聚合事务执行 |
| 非首屏分析 | 否 | 按 overscan 和滚动方向预取 |
| 词典正文、N-best、规则库、导出 | 否 | 交互时按需加载 |
| 缓存落盘、淘汰和统计 | 否 | 空闲优先级执行 |

## 7. 风险与控制

- **字符坐标漂移**：所有局部范围必须经过 source map；编辑后扩展到安全边界再重算。
- **阶段版本遗漏**：每个 artifact 类型必须集中声明版本，禁止散落字符串拼接缓存键。
- **规则更新竞态**：mutation 携带 base revision，旧结果只能被丢弃，不能覆盖新状态。
- **内存峰值上升**：段级中间产物按消费顺序释放；诊断证据不得进入正式热路径常驻。
- **渐进显示布局跳动**：改变 Token 边界的阶段必须在该范围显示前完成；后置阶段只追加不改变边界的注解。
- **过度缓存**：先测重算成本和命中率，再决定持久化层级；不得默认持久化全部完整 Token 图。
