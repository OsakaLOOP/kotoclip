# 解释目标与词典界面重构

状态：**查询协议与第二版交互重构已完成（2026-07-13）**。

## 1. 目标

阅读器不再把一个文节压缩成一次 `head_word` 查询。分析结果先生成结构化的**查询目标**，再由词典和以后接入的专用资源分别提供内容。语法规则说明保留独立的 badge 入口，不与词典气泡合并。

本模块负责：

- 决定当前整体与构词成分各自应查询什么。
- 区分严格表记查询、读音辅助排序和显式别名。
- 同时展示整体上下文与鼠标所在内部成分。
- 保留数量读法词典、专名词典等后续提供方的接入口。
- 保留文节末尾的蓝色语法 badge 及其独立说明气泡。

本模块不改变形态素、构词跨度、文节边界或表达是否成立。

词典来源的整体词汇必须在 Bunsetsu 之前恢复，不能等悬浮时临时猜测。候选生成、结构分流、跨度选择和词条绑定见 [`dictionary_lexical_units.md`](dictionary_lexical_units.md)。accepted 词典整体会直接成为本模块的可选整体面板来源。

## 2. 当前问题

现有流程为：

```text
token.bunsetsu.head_word
→ lookup_word(base_form, reading)
→ 单个 TooltipPanel
```

它丢失了已经由 B/C/D 阶段确认的结构：

- 构词整体、可变成分和固定接头／接尾不能同时表示。
- 语法规则说明与词典词条混在同一查询状态中，无法保持原有 badge 的清晰入口。
- 整体表记不存在时，通用查询会退到读音检索；`一羽` 因而可能从 `イチワ` 命中无关的 `和`。
- 内部语素只能双击后替换整个气泡，整体与内部不能对照。
- 已知词被悬浮入口直接排除，查询能力错误地依赖画像状态。

问题不在构词规则是否命中，而在命中结果没有被转换为明确的解释计划。

## 3. 对象分类

### 3.1 单一词汇

单一形态素或词法层已经确认的普通词。整体是主解释目标；没有不同的内部目标时只显示一个面板。

### 3.2 固定多语素词汇

多个形态素共同构成固定词汇，如词典中实际收录的多语素复合词。

- 整体：主词典目标。
- 内部：各原始语素仍可查询。
- 整体无精确词条时不生成整体词典面板，也不用整体读音冒充其他词条。

### 3.3 生产型构词

可变内容与接头、接尾或类型锚点组成完整词，如动作成分＋`神`、数词＋助数词、名词＋形容词接尾。

- 整体：尝试严格的整体表记查询；只有真实命中时才生成整体词典面板。
- 内部：按捕获范围分别生成内容、接头、接尾、数词和助数词目标。
- 内部目标各自查询，不把接尾释义当作整体释义。
- 规则必须声明成分角色；UI 不根据捕获名称或词性临时猜测。

例如 `一羽` 没有精确整体词条时不显示整体面板；鼠标位于 `羽` 时，内部面板只查询 `羽`，不得按整体读音 `イチワ` 返回 `和`。

### 3.4 文节语法

由阶段 C 确认的述语链、格成分、连体、连用等局部功能，以及已有 `GrammarTag`：

- `GrammarTag` 继续渲染在整个文节末尾的蓝色 badge，不进入词典双面板。
- 悬浮或点击 badge 打开独立的语法说明气泡，交互语义与旧界面一致。
- 鼠标位于文节内部时仍只处理整体词条和当前成分的词典查询。
- 局部功能或人工规则说明不伪装成词典词条。

### 3.5 跨文节表达

由阶段 D accepted 的惯用语、语法构式或呼应表达：

- 表达说明沿用表达自己的可见入口，不占用整体词典面板。
- 仅当完整表达确实命中结构化整体词条时，才可显示整体词典面板。
- 内部：当前文节中的语素或构词成分独立查询。
- 非连续表达沿用 `matched_ranges`，自由 gap 不生成表达解释目标。

多个语法或表达同时覆盖当前位置时，由各自 badge／表达入口明确选择，不按数组顺序静默改变词典查询。

## 4. 公共协议

新增独立查询规划模块，不把协议继续堆入 `DictionaryLookup`。语法 badge 仍消费 `GrammarTag`，不经过此计划。

```text
ExplanationPlan
├─ whole_candidate   已绑定整体或待验证的生产型整体目标
├─ component_targets[]
└─ default_component

ExplanationTarget
├─ target_id
├─ role              whole/content/prefix/suffix/counter
├─ provider          dictionary/quantity_reading
├─ label
├─ surface/base_form/reading
├─ morpheme_range/char_ranges
├─ lookup_policy
└─ evidence
```

`lookup_policy` 至少包括：

- `exact_form`：必须存在相同表记或结构化表记变体；读音只参与同表记结果排序。
- `kana_redirect`：纯假名允许使用词典声明的表记候选和重定向。
- `explicit_alias`：只允许规则或词典明确声明的同义表记跳转。

上下文悬浮不得使用模糊查询或“表记失败后按读音找任意词”。模糊查询只保留给用户主动搜索或词典内部导航。

解析结果独立返回：

```text
ResolvedExplanation
├─ target
├─ status            resolved/not_found/unavailable/error
├─ provider_content
├─ dictionary_lookup?
└─ match_evidence
```

accepted `DictionaryLexicalUnitAnnotation` 已绑定精确词条，不必再次猜测整体是否存在。只有生产型 WordFormation 尚无词典整体注解时才按需验证；解析为 `not_found` 时只保留内部目标，不把空结果交给 UI。日志和 CLI 仍保留未命中证据，以区别查询失败。

## 5. 规则层接入

### 5.1 构词规则

`word_formation_patterns.json` 的输出增加解释策略：

```json
{
  "explanation": {
    "kind": "compositional",
    "whole_lookup": "exact_form",
    "components": [
      { "capture": "number", "role": "content", "lookup": "exact_form" },
      { "capture": "counter", "role": "counter", "lookup": "exact_form" }
    ]
  }
}
```

`kind` 使用有限枚举：

- `lexical_whole`：固定多语素词，整体词典解释优先。
- `compositional`：生产型构词，内部成分分别查询，整体仅在精确命中时出现。

严格校验捕获存在、范围有效、角色合法且目标不重复。默认值只能用于普通单词；生产型规则必须显式声明成分。

### 5.2 文节与表达规则

- `BunsetsuFunctionAnnotation` 只提供局部功能证据，不自动构造词典查询。
- `GrammarTag` 继续通过文节末尾蓝色 badge 展示人工说明。
- accepted 表达沿用独立表达入口；pending/rejected 不进入阅读器正式解释。
- 语法／表达入口与词典浮层状态互不替换。

### 5.3 专用资源提供方

数量读法词典以后实现 `quantity_reading` 提供方，输入数值捕获、助数词捕获及上下文，输出读音、变音证据和来源。它只补充解释内容，不回写形态分析读音，也不改变构词是否成立。

## 6. 后端职责

新增：

```text
crates/kotoclip-core/src/explanation/
├─ planner.rs         从 B/C/D 注解生成解释计划
├─ resolver.rs        调度各内容提供方
└─ mod.rs
```

职责边界：

- `planner` 是纯逻辑，可在没有外部词典时测试。
- `resolver` 使用 `DictionaryEngine` 检查精确表记并取得内容。
- `DictionaryEngine` 增加带策略的查询入口；现有 `lookup_word` 保留给主动搜索和内部导航。
- Tauri 新增一次面向 token 的解释命令，不让前端根据 `head_word`、规则 ID 或捕获名自行决定查询字符串。
- 紧凑分析 IPC 不携带词典内容；悬浮时按需解析，并按 `target_id＋词典优先级` 缓存。

第一版命令可以是：

```text
resolve_token_explanations(token, focus_morpheme, subject_id?, priority_list)
→ ExplanationBundle { plan, whole?, focused_component }
```

同一 token 内移动鼠标时复用 `plan` 和可选整体结果，只请求尚未缓存的内部目标。

## 7. 双面板交互

旧 `TooltipPanel` 拆为：

- `ExplanationPopover`：定位、悬浮通道和一至两个词典面板的编排。
- `ExplanationPanel`：统一内容外壳。
- 现有 `DictionaryContent`、选项条和词典导航继续复用。
- 语法 badge 及说明组件保留为独立组件，不嵌入 `ExplanationPopover`。

### 7.1 面板关系

- 整体面板只表示已经精确命中的整体词条；未命中时不存在。
- 内部面板表示鼠标当前所在成分。
- 两者分别显示“整体”和“内部 · 内容／接头／接尾／助数词”等短标签。
- 原文中的当前内部范围使用与内部面板一致的弱强调；不增加永久标记。
- 当内部目标与整体完全相同，不显示重复面板。

### 7.2 鼠标行为

- 悬浮文节内部后打开当前成分面板；若聚合整体有精确词条，再同时打开整体面板。已知／未知状态不影响查询。
- `pointerenter` 具体语素或捕获范围时更新内部面板，不再要求双击。
- 从正文移动到任一现有面板时保持浮层；离开正文与整个浮层后统一延迟关闭。
- 内部查询带请求序号和目标缓存，旧请求不能覆盖新目标。
- 鼠标位于文节末尾蓝色 badge 时打开该 badge 的语法说明气泡，不把词典面板切换为规则内容。
- 没有整体词条时，单个内部面板以当前语素范围为锚点；跨语素移动时允许面板随新锚点重新定位。
- 存在整体词条时，整体与内部面板以整个文节范围为共同锚点；跨语素移动只更新内部内容，不移动浮层组。

### 7.3 定位与裁切

主布局面向 16:9 桌面视口。整体面板存在时，两个词典面板以文节 `DOMRect` 作为一个浮层组参与布局，不能各自独立定位；整体面板不存在时，内部面板以当前语素 `DOMRect` 使用相同的定位顺序：

1. 双面板强制左右并排、顶部对齐，作为不可拆分的浮层组。
2. 先比较锚点上、下的真实可用高度；一侧能容纳目标高度时选择该侧，两侧状态相同时选择空间更大的一侧。
3. 再以锚点中心对齐面板组；接近视口左右边界时只整体水平平移，不改变面板关系。
4. 面板高度上限为 `480px`；空间不足时进一步收紧到该侧可用高度，两个面板独立滚动并共享高度上限。
5. 极窄视口目前只将两个面板等比收窄以防溢出，不自动改为上下分置；墨水屏样式属于测试入口，正式降级布局另行设计。

布局计算输入使用实际 `DOMRect` 和测量后的面板固有尺寸。窗口缩放、滚动和虚拟列表复用时重新定位。

## 8. 典型结果

| 文本 | 可选整体词典面板 | 内部词典面板 |
| --- | --- | --- |
| `警察署`（固定多语素） | 精确命中时显示 `警察署` | 随鼠标查询 `警察` 或 `署` |
| `冷やし神` | 只有精确收录时显示 | 随鼠标查询 `冷やす` 或 `神` |
| `煙草臭い` | 只有精确收录时显示 | 查询 `煙草` 或 `臭い` |
| `第一話` | 只有精确收录时显示 | 查询 `第`、`一` 或 `話` |
| `一羽` | 未收录则不显示 | 位于 `羽` 时只查询 `羽`，不返回 `和` |
| `座っている` | 只有精确收录时显示 | 查询 `座る` 或 `いる`；语法说明仍在文节末尾 badge |
| `耳を傾ける` | 只有结构化整体词条精确命中时显示 | 查询当前位置内部成分；表达说明走独立入口 |

## 9. 实施顺序

1. 固化上述对象分类及 `ExplanationPlan` 单元测试，加入 `一羽→和` 反例。
2. 为构词 schema 增加解释策略并迁移现有规则。
3. 实现严格表记查询和解释解析器；CLI 输出计划、落点与未命中原因。
4. 接入 Tauri 与 TypeScript 协议，保持现有主动查词入口兼容。
5. 重建双面板浮层，移除已知词限制和双击查语素路径。
6. 用代表性语料验证整体、内部、规则主题、裁切和请求竞争。
7. 更新真实章节审计与本追踪文档；数量读法提供方单独实施。

## 10. 验收条件

- 每个正式构词都能生成明确的整体候选和内部查询目标；语法／表达保持独立说明入口。
- `一羽` 整体缺词条时不会因读音命中 `和` 或其他无关表记。
- 固定多语素词同时保留整体与内部查询。
- 生产型构词的内容、接头和接尾分别可查，不用接尾释义代替整体。
- 文节末尾蓝色语法 badge 保持原有位置和说明职责，不依赖词典是否命中。
- 已知词与未知词具有相同的解释入口。
- 具体语素随鼠标更新，不再依赖双击。
- 整体精确命中且空间允许时两个面板横向并列；未命中时只显示内部面板；任何窗口尺寸下均不越界或遮住锚点。
- 内部目标快速切换时无旧结果覆盖、明显抖动或重复查询。
- 后续数量读法词典可以作为新提供方接入，无需修改构词、文节或表达模型。

## 11. 已实现交接

- 阅读器通过 `DictionaryLexicalUnitAnnotation` 判断整体面板是否存在，不再对任意 head word 猜整体词条。
- 有整体词条时，整体与内部面板作为一组按文节定位；内部语素变化只更新内部内容。
- 无整体词条时，单面板按当前语素定位并随语素移动。
- 窄窗口把整体放在文节上方、内部放在下方，避免横向裁切。
- 已知词不再禁止悬浮查询；双击语素入口已移除。
- 文节末尾蓝色语法 badge 仍使用原有 `GrammarTag` 说明，不转成整体词典面板。
- 胶囊级完整释义和导出通过统一 `dictionaryTargetForToken` 选择目标：精确词典整体优先；生产型构词回退到规则词头语素。
- `一羽` 没有精确整体注解，因此不会以 `イチワ` 查询并显示 `一和`；内部可分别查询 `一` 和 `羽`。

## 12. 第二版交互实现

第一版遗留的问题不在查询对象，而在前端把 DOM 事件、查询请求、双面板状态和布局集中到段落 `mouseover` 处理器。第二版按以下边界重构：

```text
DOM 语义命中
→ ExplanationSession
→ 整体／内部独立请求通道
→ ExplanationPopover
→ 实测尺寸布局
```

### 12.1 聚焦与切换

- 形态素节点已有稳定的 `data-morpheme-index`，因此语素身份继续由 DOM 语义命中确定；不引入全局坐标扫描。
- 段落使用委托的 `pointerover/pointerout`，同时解析 `target` 与 `relatedTarget`，只在语义 key 真正变化时转换状态。
- 同文节跨语素立即更新内部目标；跨文节立即建立新会话；同一目标内部的子节点变化不重复查询。
- 只有正文与浮层之间的物理缝隙使用 140ms 关闭宽限。定时器不参与打开、目标切换或查询提交。
- 整体与内部各自持有请求代次、历史和加载状态；旧请求不能写入新目标，整体导航也不能覆盖内部状态。
- 查询结果按 `word＋reading` 缓存；同一目标再次进入时同步恢复，不重复发起 IPC。

### 12.2 组件边界

- `useExplanationSession` 负责会话、请求竞争、缓存、关闭宽限和锚点刷新。
- `ExplanationPopover` 只编排整体与内部面板，并监听不受外框高度约束的内容层尺寸变化。
- `TooltipPanel` 降为单面板内容外壳，不再包含语法说明或决定相邻面板位置。
- `GrammarPopover` 独立展示 `GrammarTag`；badge 恢复指针命中，已知词不再隐藏语法入口。
- `hitTest` 将 DOM 节点归一化为 `morpheme/token/grammar/panel/outside`，阅读器不再直接比较任意 DOM 子节点。

### 12.3 布局与回归

- 定位使用文节／语素 `DOMRect` 与 `ResizeObserver`。面板宽度取外框实测值，高度取 `scrollHeight`；观察内部内容层，不观察受 `max-height` 约束的滚动外框，避免排版结果反向改变下一轮布局输入。
- 单、双面板统一先确定上／下方向，再水平平移到安全边距内；双面板始终左右捆绑，不参与上下分置评分。
- 滚动和窗口缩放刷新锚点；虚拟列表已卸载锚点时关闭会话。
- `140ms` 关闭宽限先经过几何桥接判定，只覆盖正文与面板、双面板之间的实际窄缝；从卡片上沿或外侧离开时立即关闭，不再形成随移动速度扩大的隐形保持区域。
- `npm run test:ui` 覆盖单面板视口边界、16:9 双面板捆绑与高度限制、水平边缘平移、窄屏安全收窄、跨语素／跨文节转换语义，以及面板上沿离开和双面板窄缝穿越。
- `npm run build` 同时执行 TypeScript 类型检查和生产构建。
