# Kotoclip 产品目标、发布路线与自适应阅读架构

状态：统一口径草案；实现基线已审计至 2026-07-19

范围：产品目标、发布边界、模块设计、数据架构、分析算法与评测口径。

本文档整合当前实现、产品方向和发布约束。它不是固定不变的功能清单；涉及记忆、干预和阅读效果的算法必须经过真实使用和评测后迭代。

## 1. 产品定义

Kotoclip 是一个本地优先、文本驱动的自适应日文沉浸阅读系统。

系统以用户真正想读的文本为唯一主场景，识别其中的词汇、文法、表达、句法和篇章现象，并根据用户历史、遗忘风险、当前理解状态和阅读模式，动态决定辅助信息的显隐、强度、重复方式和 AI 介入程度。

产品的普惠目标是降低从教材和考级学习进入真实日文阅读的门槛。一个约 N3 水平、尚不能独立阅读生肉的用户，应当能够选择难度适当的现代日文文本，在不中断主阅读流程的辅助下完成第一本全书，并将这次阅读转化为可持续的词汇、文法和阅读能力积累。

“无障碍”指阻碍能够在正文内通过默认提示、简单鼠标操作或快速提问解决，不指隐藏原文难度、自动给出完整翻译，也不承诺任意用户可以无辅助阅读任意文学或专业文本。

Kotoclip 不是：

- 从零开始教授日语的完整课程软件；(e.g. duolingo, 新标日app)
- 以 JLPT 刷题, 背词为中心的备考产品；(e.g. MoJi辞书)
- 需要用户手工制卡的传统卡片工具；(anki及其社区)
- 必须联网、必须登录或必须使用 AI 的云阅读器；(现有通用chatbot)
- 将所有分析结果永久铺满正文的标注展示器。

核心流程是：

```text
真实文本
  → 本地语言分析
  → 统一文本现象
  → 个性化辅助投影
  → 阅读行为与主动回忆
  → 可重算用户状态
  → 动态卡片与下一次阅读
```

### 1.1 两种产品目标

同一篇文本可以承担两种不同任务，不能用同一套默认调度策略处理：

| 目标     | 用户意图                                 | 系统优先级                               | 典型结果                         |
| -------- | ---------------------------------------- | ---------------------------------------- | -------------------------------- |
| 沉浸阅读 | 我想把这本书读下去                       | 连续性、低中断、快速消除局部障碍         | 完成章节和全书，顺便积累知识     |
| 文本教材 | 我想用这篇文本系统完成 N2 文法和读解能力 | 目标覆盖、显式讲解、回忆、测评和缺口追踪 | 覆盖目标现象，识别不足并安排复习 |

两种目标共享语素、词典、文节、表达、读解、知识项、用户事件和卡片底层；差异由 `SessionMode` 或等价的策略层控制，包括辅助密度、是否主动打断、目标现象排序、讲解深度、提问频率和完成条件。用户可以按文本或会话切换模式，不能要求一个全局模式同时满足两个相反目标。

“任意文本作为教材”应分成三个承诺层级：

1. **文本可分析**：任意可处理的现代文本都可以获得词汇、文法、表达和读解辅助；
2. **文本可教材化**：系统根据 N2 目标矩阵标出已出现、未出现、已理解和待复习的现象，并生成学习路径；
3. **规则体系足以完成 N2**：只有经过体裁、覆盖、难度、顺序和测评验证的规则用于文本集合才能做此承诺，单一本书不能保证覆盖全部 N2 文法和读解能力, 但应通过完整的规则覆盖, 主动设计引导或辅助读者选择适合自己的文本阅读。

因此，用户可以任意挑选文本作为个人教材，但系统必须明确显示“这本书教到了什么、还缺什么”，而不是把一次分析结果包装成完整 N2 课程。

## 2. 核心用户与价值

### 2.1 主要用户

主要面向约 N3 至 N1 及以上、希望阅读真实日文材料的用户，包括尚未成功读完第一本生肉、受到词汇、文法、语境和断句阻碍的学习者，也包括已经能够独立阅读但仍受到低频知识和遗忘干扰的长期读者。

完成初级教材、能够在帮助下理解适当难度文本的用户也可以使用。系统不设置绝对等级门槛，而是通过文本难度、辅助密度和阅读模式适配能力差异。

N1 不是终点。高级用户仍需要处理低频词、特殊义项、文学表达、搭配、篇章结构和长期遗忘。

### 2.2 即时价值

用户第一次使用时，应当能够：

- 不输入汉字或猜测读音即可快速查词, 查词, 通过简单鼠标操作取得读音、释义和局部解释；
- 不离开正文即可区分词汇边界和前后文节, 理清语法规则和实际出现情况, 进而初步理解文本逻辑；
- 通过简单鼠标操作, 在本地结果不足时快速向 AI 提问, 或者触发注释和规则的修正, 省略内容的补全；
- 尽量不在词典、聊天机器人和原文之间, 因为 UI 变化或者认知负荷而需要切换注意力；
- 循序渐进地完成一段、一章并最终完成一本真实文本，且完成一定的积累和学习。

首要用户结果不是“查到了更多词”，而是原本无法持续阅读的用户第一次在原生日语环境完整读完自己真正想读的书, 且在辅助下有一定收获。

### 2.3 长期价值

系统长期积累：

- 用户读过的词汇、义项候选、文法和表达；
- 每个知识点的来源、例句、N1-5水平、出现频率和跨文档复现；
- 主动查阅、人工纠正和回忆结果；
- 动态生成并更新的阅读卡片；
- 用户的遗忘风险和辅助需求。

随着历史增加，系统应当减少不必要提示，并更准确地强化真正薄弱的内容，使用户能够更自信、流畅和深入地阅读轻小说、现代文学、网页和字幕等真实材料。

## 3. 最高设计原则

### 3.1 沉浸优先

学习调度必须服从阅读沉浸约束。

```text
最大化：长期理解与记忆收益

约束：
- 当前阅读的中断预算
- 用户选择的阅读模式
- 当前场景的连续性
- 辅助密度上限
- 系统判断的置信度
```

用户看不懂时会自然回到辅助层。系统不应为了追求理论上的最佳复习时间而强制破坏阅读。

### 3.2 本地确定性核心

以下能力必须离线可用：

- 文本准备和范围管理；
- 形态素、读音、文节、表达和规则匹配；
- 本地词典适配器；
- 用户画像、阅读事件和卡片；
- 基础动态显隐；
- 数据导出和备份。

本地规则、模型和算法构成规范事实层。AI 是可选增强，失败时只能降低解释深度，不能阻断阅读。

### 3.3 私有数据与可迁移性

- 用户文本、阅读历史、画像和私人例句默认只保存在本地；
- 未来云服务以 AI、加密备份和同步为增强层；
- 同步应采用端到端加密设计；
- 用户可以完整导出和删除数据；
- 取消云服务后，本地阅读和历史仍然可用；
- 1.0 不建设公开卡片社区。

### 3.4 不确定性必须保留

分词、义项、文法、句法和篇章分析都可能不确定。

- 不确定结果以候选、证据和置信度表达；
- 不把模型输出伪装成唯一事实；
- 用户纠正优先于自动分析器；
- 所有纠正应可撤销；
- 派生状态必须能够在算法或目录升级后重算。

### 3.5 覆盖与准确分离

技术预览可以只覆盖目标知识体系的 15% 至 30%，但已默认展示的结果必须相对可信。1.0 才同时追求目标域内 90% 至 95% 的目录覆盖与真实现象召回。

不得用增加低质量规则换取表面覆盖率。

## 4. 发布层级

### 4.1 技术预览版

目标：证明语言分析和知识目录可以稳定扩展。

范围：

- 可靠的本地查词、读音、词汇边界和文节；
- N1 至 N5 元数据接入；
- 词汇、文法、文型和表达的统一目录；
- 每个模块约 15% 至 30% 的目标目录覆盖；
- 高精确率的已验证规则；
- 人工纠正和审计工具；
- GiNZA 作为可选结构分析验证组件。

技术预览不承诺完整读解理论、成熟遗忘算法或自动强制复习。

### 4.2 阅读测试版

目标：证明真实文本中的阅读辅助和历史聚合有效。

新增：

- 基础读解表现、文型和接续功能；
- 文档、章节、阅读位置和历史；
- 统一知识项和文本实例；
- 语义学习事件；
- 动态卡片生成与更新；
- 基础遗忘风险；
- 多级辅助显隐；
- BYOK AI 提问和上下文讲解。

### 4.3 v1.0

目标：形成可验证的完整闭环。

```text
低中断首次阅读
  → 自动收集知识点和出处
  → 动态更新卡片
  → 记录回忆与纠正
  → 下一次阅读调整辅助
```

1.0 要求：

- 基础语法高精确率和高文节边界准确率；
- 现代标准日语目标域内主要模块达到 90% 至 95% 的目录覆盖或现象召回目标, 部分数据无法判断最优选可以适当交给用户或 ai 选择；
- 减少严重的语法, 词汇, 表达误匹配问题, 小于5%;
- 基础读解与动态卡片闭环完成, 高度可读, 结构样式清晰；
- 用户能够完整导出知识项、出处、历史和卡片状态；
- AI 不可用时基础阅读完整可用。

完整篇章推理、成熟自动打断策略、全面 AI 测评和古典日语不作为 1.0 必要条件。

## 5. 1.0 文本域边界

正式评测以现代标准日语为主：

- 轻小说和一般现代小说；
- 《斜阳》等近现代文学；
- 中长篇新闻、评论和说明性网页；
- 可信字幕组的影视 SRT 字幕, galgame 提取的脚本；
- 专名、拟声词、口语、省略表达等边界压力专项文本。

以下内容作为实验支持，不进入现代日语 1.0 的高覆盖承诺：

- 古典日语；
- 方言密集文本；
- 固有 OCR 错误, 转录错误的文本；
- 高度专业的法律、医学和工程文本；
- 高度实验性文学。

古典日语以后通过独立 `LanguageProfile` 接入，使用独立词典、活用体系、规则目录和评测集，不向现代日语规则库持续添加例外。

## 6. 当前实现基线

截至 2026-07-19，当前工程优势仍集中在文本分析、长文档基础设施、语法工程框架和多词典查询。下表的百分比表示“相对 v1.0 发布闭环的完成度”，不是代码量或单模块工程完成度。

| 模块 | 当前状态 | v1.0 阶段判断 |
| --- | --- | ---: |
| 长文本会话、渐进加载、Patch、缓存 | 增量管线 P1～P6 重构完成；首批、暖缓存、mutation 和诊断入口均已建立 | 90% |
| Vibrato 形态素、真实 N-best、范围 | 真实 lattice N-best、稳定范围和人工选择已实现；语境重排与作用域仍待深化 | 80% |
| 多词典查询与气泡 | 三词典适配器、统一 occurrence／sense IR、查询证据、单活动词典气泡和 18 词审计已完成；进入样本驱动扩展 | 75% |
| 词汇整体、构词、文节 | 正式管线与前三话基线存在，但目录仅有 6 条构词规则、4 条词典词汇候选模式和 10 条边界规则 | 60% |
| 表达识别引擎 | 连续、非连续、用户规则、预演和类型化候选已实现；341 项人工审计、结构证据和 span graph 未完成 | 55% |
| 语法识别与讲解工程框架 | G0～G5 闭环完成；当前 105 concept、154 sense、112 rule／realization、105 份讲解，正文精确解析和目录审计可用 | 80% |
| 语法／表达目录与 N1～N5 覆盖 | 当前 concept 只覆盖 N3 6 项、N4 55 项、N5 43 项和 1 项未知；N1/N2、统一目标目录分母和多体裁召回尚未建立 | 20% 至 30% |
| 文档库、章节、导入与阅读位置 | 当前主界面仍以粘贴文本为入口；没有正式文档库、章节实体、导入适配器和恢复位置 | 5% |
| 句法提供者与基础读解 | 仅有文节功能和 `syntax_evidence` 预留字段；没有正式 provider、依存／小句基线或读解解释模块 | 0% 至 5% |
| 用户画像 | 具备词汇曝光、已知／未知、汉字推断、分词和词典选择；状态仍以覆盖计数为主 | 25% |
| 语义事件、知识聚合与完整导出 | 没有 `learning_events`、`review_logs`、知识项、文本实例和事件重放；现有导出仅覆盖本次选中词汇 | 5% 至 10% |
| 动态卡片与遗忘调度 | 没有正式卡片生命周期、复习队列和 FSRS 状态 | 0% 至 5% |
| 动态阅读辅助 | 当前 `master` 没有阅读模式策略、干预等级或密度预算；`backup-master` 上的原型提交不计入当前完成状态 | 0% 至 5% |
| AI 安全协议与实际调用 | 词典消歧证据 schema 和宿主注入端口存在；没有 HTTP transport、Tauri 命令、凭据、UI、评测或通用阅读问答 | 15% 至 20% |

当前第一话研究结果中的 95.73% 是词汇出现级词典查询覆盖，不代表词义、读音、边界、文法或解释均达到 95%。

按可独立实现和验收的工作包计，v1.0 仍有 8 个模块包：统一协议与评测、语言覆盖收口、文档库与发布资源、句法与基础读解、事件与知识聚合、动态卡片与 FSRS、自适应阅读辅助、BYOK AI 问答。外部分析器、深度文本分析、FSRS 实现库、凭据存储和 AI 网络层均先经过条件性实验，不因进入候选清单而默认成为依赖。完整清单、规模、依赖和提交拆分见 [`v1_completion_plan.md`](v1_completion_plan.md)。

## 7. 总体模块设计

```text
Document Core
├─ Source Adapter Protocol
├─ Text Preparation
├─ Morpheme and Lexical Pipeline
├─ Expression and Grammar Pipeline
├─ Syntax Provider Layer
├─ Discourse and Reading Theory Layer
├─ Text Phenomenon Registry
├─ Dictionary Adapter Layer
├─ User Data and Event Store
├─ Knowledge and Card Engine
├─ Intervention Scheduler
├─ AI Assistant Port
└─ Export, Backup and Sync
```

### 7.1 文本入口

核心只接收规范文档格式。外部插件、skill 或 agent 负责将 EPUB、网页和其他来源转换成该格式。

1.0 正式入口：

- 粘贴文本；
- TXT；
- Markdown；
- SRT；
- 基础文档库、章节和阅读位置。

规范文档应保留：

- 来源和标题；
- 章节与段落；
- 对白或字幕时间轴；
- 稳定字符位置；
- 可选原始来源标识；
- 文档版本。

### 7.2 统一文本现象

词汇、文法、表达和篇章关系不能继续通过各自 UI 类型孤立扩展。建议引入统一协议：

```text
TextPhenomenon
- phenomenon_id
- catalog_entry_id
- category
- subtype
- char_range
- morpheme_range
- token_range
- normalized_form
- jlpt_level
- frequency_band
- register
- confidence
- evidence
- explanation_targets
- prerequisite_ids
- conflict_group
```

识别结果与用户状态必须分离：

```text
PhenomenonOccurrence
- occurrence_id
- document_revision_id
- phenomenon_id
- char_range
- analyzer_id
- analyzer_version
- evidence
- confidence
```

### 7.3 知识目录

每个目录条目应包含：

- 规范 ID；
- N1 至 N5 或未知等级；
- 等级来源；
- 类别和子类；
- 接续、语义和语用；
- 语域；
- 正例和反例；
- 相近现象；
- 匹配规则；
- 解释模板；
- 来源、版本和许可；
- 审计状态。

状态分为：

```text
verified
sourced
experimental
user
deprecated
```

AI 可以协助结构化、生成候选和构造测试，但不能直接将条目升级为 `verified`。

### 7.4 词典层

Kotoclip 发布词典适配器和统一查询协议，不分发当前研究用大辞林内容。

词典类型：

- 可合法分发的基础词典；
- 用户自行导入的本地词典；
- 未来单独授权的商业词典；
- 未来 SaaS 授权查询；
- 仅用于研究、不得分发的开发词典。

词典来源、许可、版本和导出限制必须进入元数据。

## 8. 句法和主干来源

### 8.1 GiNZA 验证路线

技术预览使用 GiNZA 标准模型作为可选本地 sidecar，验证：

- 文节边界；
- 文节主辞；
- 依存边；
- 谓语及直接论元；
- 小句和小句主辞候选。

当前 Vibrato/IPADIC 管线继续拥有规范语素、读音、N-best 和词典链路。GiNZA 不直接改写 Token。

```text
原始字符范围
├─ Vibrato/IPADIC Artifact
└─ GiNZA SyntaxArtifact
       ↓
字符范围对齐
       ↓
联合证据与候选结构
```

### 8.2 KWJA 验证路线

KWJA 作为更深层研究提供者，重点验证：

- 述语项结构；
- 桥接照应；
- 共指；
- 篇章关系。

其依存分析可以作为强候选，但篇章关系准确度明显低于依存，不得作为无审计的确定事实。

### 8.3 所有权边界

- `DocumentSession` 保持规范文本和稳定范围；
- 外部分析器输出版本化 Artifact；
- 多分析器一致时提高置信度；
- 冲突时保留来源和候选；
- 用户纠正优先；
- 联合决策层决定是否影响下游；
- 模型升级只重建 Artifact，不破坏历史知识项和卡片身份。

相关外部来源见 [sources.md](../sources.md)。

## 9. 读解理论层

教材作为知识体系和分类参考，Kotoclip 将其重新建模为结构化、文本分析驱动的现象，不复制教材的教学呈现形式。

文本教材模式需要额外维护：

- N2 文法、文型、接续和读解表现目标矩阵；
- 当前文本对每个目标的出现位置、语境、难度和解释状态；
- 用户是否只是看过、是否理解、是否能在新语境中识别和运用；
- 当前文本无法覆盖的目标，以及需要其他文本或专门练习补足的缺口；
- 文本内自然阅读、显式讲解、回忆问题和章节总结之间的关系。

这使“文本教材”成为目标驱动的学习会话，而不是把所有标注永久显示在正文中。

### 9.1 局部读解现象

- 判断与推量；
- 限定与强调；
- 对比与让步；
- 原因与结果；
- 条件与假设；
- 目的与手段；
- 引用与传闻；
- 评价与态度；
- 视点和立场；
- 程度、范围和数量关系。

### 9.2 接续和篇章功能

- 因果；
- 转折；
- 追加；
- 并列；
- 对比；
- 换言；
- 例示；
- 总结；
- 话题转换；
- 让步；
- 结果；
- 推论。

### 9.3 篇章关系

```text
DiscourseRelation
- source_span
- target_span
- relation_type
- connective
- explicit_or_implicit
- confidence
- evidence
```

初期优先提供对阅读有直接价值的谓语、从句、修饰、引用、接续和段落功能。省略、指代和篇章关系允许保留多个候选，不追求每句唯一解析。

## 10. 本地大数据与事件架构

### 10.1 结论

单用户长期数据不需要分布式大数据平台。SQLite/rusqlite 适合设备本地、低写入并发的事件和画像存储。

系统采用轻量事件溯源：

```text
不可变事实层
├─ 文档和版本
├─ 文本现象实例
├─ 阅读会话
├─ 语义学习事件
└─ 正式复习记录

可重算投影层
├─ 曝光汇总
├─ 用户知识状态
├─ 卡片状态
├─ 遗忘风险
└─ 文档和体裁统计
```

当前 `exposure_history` 可以继续作为过渡投影，但不能继续作为唯一事实来源。

### 10.2 建议表

```text
documents
document_revisions
reading_sessions
catalog_entries
knowledge_items
knowledge_item_links
phenomenon_occurrences
learning_events
review_logs
user_knowledge_states
card_states
analysis_artifacts
projection_checkpoints
sync_operations
```

### 10.3 语义事件

```text
LearningEvent
- event_id
- occurred_at
- session_id
- document_id
- document_revision_id
- occurrence_id
- knowledge_item_id
- event_type
- intervention_level
- duration_ms
- result
- analyzer_version
- payload_schema_version
- payload_json
```

记录：

- 知识点真正进入可见范围；
- 用户主动查词；
- 展开完整释义；
- 请求 AI；
- 标记已知或未知；
- 修正分词、读音、义项或规则；
- 主动回忆成功或失败；
- 跳过、推迟或关闭卡片；
- 用户切换辅助模式。

不直接记录每次鼠标移动、滚动像素或 hover 抖动。高频 UI 数据先在内存中聚合，只写入具有学习语义的结果。

### 10.4 派生画像

```text
UserKnowledgeState
- knowledge_item_id
- exposure_count
- distinct_document_count
- lookup_count
- successful_recall_count
- failed_recall_count
- last_seen_at
- last_reviewed_at
- memory_stability
- memory_difficulty
- retrievability
- next_due_at
- projection_version
```

算法升级时：

```text
learning_events + review_logs
  → 重放或增量投影
  → user_knowledge_states
  → card_states
```

### 10.5 保留、删除和压缩

- 默认长期保留有语义的原始事件；
- 汇总、掌握度和卡片时间均可删除并重建；
- 用户可以删除文档、某类行为或全部历史；
- 删除文档时允许保留匿名知识统计，也允许彻底级联删除；
- 旧的高频非关键事件可以压缩成日级或会话级摘要；
- 每次投影记录算法和 schema 版本；
- 迁移前创建可恢复备份。

### 10.6 性能策略

- SQLite 使用 WAL 和短事务；
- 阅读热路径只做批量事件写入和索引查询；
- 投影更新可增量执行；
- 大规模重算在后台或显式维护任务中运行；
- 分析 Artifact 与用户事件分库或分表管理；
- 大型研究统计导出到 DuckDB/Polars，不进入应用热路径；
- 全文搜索使用 FTS5；
- 当前不依赖向量数据库。

### 10.7 同步和备份

未来 SaaS 同步以版本化操作日志为基础：

- 本地数据库仍是完整数据源；
- 云端存储端到端加密数据块；
- 用户纠正和复习记录具有稳定事件 ID；
- 同步冲突不通过最后写入静默覆盖；
- 派生投影默认不必同步，可以在目标设备重建；
- 大型文档和模型 Artifact 与画像事件分开同步；
- 支持完整导出、恢复和云端删除。

## 11. 知识项聚合与动态卡片

### 11.1 三层模型

```text
规范知识项
  → 文本中的具体出现
  → 当前复习策略生成的卡片视图
```

卡片不是底层知识实体。

### 11.2 词汇层级

首版不强制在“按词条”与“按义项”之间二选一：

```text
LexemeCluster：辞书形 + 读音 + 词性
  ├─ SenseCandidate A
  ├─ SenseCandidate B
  └─ 未消歧出现
```

- 卡片默认可以绑定 `LexemeCluster`；
- 当前语境首选义项突出显示；
- 其他义项折叠保留；
- 有充分证据时建立义项级状态；
- 无法消歧的实例不能被强制写入某个义项；
- 未来可以研究多义项聚合呈现。

### 11.3 文法和表达

文法与表达绑定规范目录 ID，每次真实出现作为独立实例，保留：

- 原句；
- 精确范围；
- 槽位映射；
- 文档和章节；
- 分析器和规则版本；
- 人工纠正。

### 11.4 卡片生命周期

```text
发现知识项
  → 聚合历史实例
  → 选择适合的私人例句
  → 生成卡片视图
  → 用户回忆或继续阅读
  → 更新状态
  → 后续实例替换或补充例句
```

私人例句来自用户实际文本，只在本地和加密备份中存在。公共预制例句必须来自公版、原创、开放许可或正式授权来源。

### 11.5 导出

1.0 至少支持 UTF-8 JSON 和 CSV，导出：

- 稳定知识项 ID；
- 辞书形、读音、词性和义项候选；
- 文法或表达目录 ID；
- 例句和精确高亮范围；
- 文档、章节和出处；
- 首次及最近出现时间；
- 出现次数；
- 用户纠正；
- 掌握和复习状态；
- schema 版本。

不得导出无权复制的完整词典正文。AnkiConnect、APKG 或插件同步建立在稳定导出协议之上，不决定内部 schema。

## 12. 遗忘和调度算法

### 12.1 FSRS 的位置

Rust `fsrs` crate 可用于：

- 正式卡片的记忆状态；
- 复习间隔；
- 可回忆概率；
- 基于复习历史的参数优化；
- 工作量模拟。

FSRS 只能消费真实的主动回忆结果。自然看到、弱提示或悬浮查词不能伪装为 `Good` 复习。

### 12.2 两类信号

```text
阅读信号
- 自然出现
- 主动查词
- 停留和回看
- AI 请求
- 同一知识点跨文本复现

复习信号
- 主动回忆
- 选择或输入答案
- 成功、迟疑或失败
- 响应时间
```

阅读信号影响“是否需要辅助”和“是否值得生成卡片”；复习信号进入正式遗忘调度。

### 12.3 冷启动

在积累足够复习记录前：

- 使用默认 FSRS 参数；
- 结合用户自报等级、词汇频率和 JLPT 等级；
- 使用主动查词和重复查阅估计困难度；
- 不把推断状态当成已验证掌握；
- 达到足够真实复习记录后再进行个性化参数优化。

### 12.4 动态干预调度

FSRS 决定“何时值得复习”，阅读调度器决定“现在是否适合介入”。

```text
InterventionScore = f(
  retrievability,
  knowledge_importance,
  comprehension_risk,
  future_occurrence_opportunity,
  lookup_probability,
  interruption_cost,
  current_density,
  confidence
)
```

调度器输出：

| 等级 | 阅读行为              |
| ---: | --------------------- |
|    0 | 完全隐藏辅助          |
|    1 | 极弱图形提示          |
|    2 | tag、下划线或边缘标记 |
|    3 | 悬浮或点击时优先显示  |
|    4 | 自动显示短提示        |
|    5 | 整合上下文解释        |
|    6 | 文中可跳过的回忆检查  |
|    7 | 半屏重点复习          |
|    8 | 全屏卡片式处理        |

1.0 只需证明动态卡片生成、历史更新和基础多级显隐。自动半屏/全屏打断必须经过后续教育学研究和真实数据验证。

### 12.5 阅读模式

- 沉浸阅读：以完成文本为目标，严格限制主动提示和中断；
- 平衡阅读：允许弱提示和少量自然回忆；
- 文本教材：围绕 N2 目标现象提高讲解、标记、提问、回忆和缺口追踪密度；
- 强化阅读：用户明确希望提高学习密度时，增加结构化回忆和卡片处理；
- 主动复习：用户明确离开正文，进入卡片或目标知识项处理流程。

场景信号可以降低干预，例如连续对白、快速滚动、章节高潮和用户频繁跳过提示。用户模式始终高于自动调度。

## 13. AI 模块

### 13.1 1.0 发布模式

- 用户自带端点/API key；
- 支持可替换提供者；
- 不建设官方账号、计费和代付服务；
- 请求由用户授权；
- 默认只发送当前句、必要上下文和本地证据；
- 结果本地缓存；
- AI 不可用时降级到本地能力。

长期可提供官方 SaaS，但仍保持本地完整数据源、端到端加密备份和可关闭同步。

### 13.2 AI 任务

- 词典义项消歧；
- 当前语境短解释；
- 文法槽位映射；
- 句子骨架和修饰解释；
- 指代和省略候选；
- 接续与段落作用；
- 阅读理解提问；
- 用户答案评价和错误归因。

### 13.3 约束

- 必须引用当前文本和本地证据；
- 区分事实、候选和推断；
- 允许弃权；
- 不直接改写本地规范状态；
- 不直接把一次 AI 评分写成掌握事实；
- 失败或超时不能阻断阅读。

## 14. 评测体系

每个模块分别报告：

- 目录覆盖率；
- 真实现象召回率；
- 精确率；
- 边界准确率；
- 分类准确率；
- 解释可用率；
- 用户价值覆盖率；
- 严重错误率。

低中断阅读还应记录本地指标：

- 从指向文本到有效释义的延迟；
- 一次查词所需操作数；
- 是否切换到外部应用；
- 因边界或读音错误造成的重复尝试；
- 每千字主动查词和误触次数；
- 长停顿和回看；
- 章节完成情况；
- 用户自评流畅度、理解度和疲劳度。

产品主导者的主观阅读体验决定是否继续打磨；行为指标用于定位原因、比较版本和防止局部优化制造新的打扰。

## 16. 已修正的不现实假设

- 不把一项 95% 指标代表为完整语言理解准确度；
- 不要求首版同时支持所有文本体裁；
- 不把古典日语混入现代规则库；
- 不从零实现依存和主干模型，先验证 GiNZA/KWJA；
- 不让外部分析器直接改写规范 Token；
- 不把累计曝光当成完整用户画像；
- 不把自然看到当成一次成功复习；
- 不在缺少真实数据时自动强制打断阅读；
- 不把卡片当成底层知识实体；
- 不要求首版完成义项唯一消歧；
- 不分发权利不明的研究词典；
- 不建设公开私人例句社区；
- 不为单用户数据引入分布式大数据平台；
- 不让 AI 成为阅读核心的单点故障；
- 不让未来 SaaS 取消本地完整能力。
